Q1: Kafka、RocketMQ、RabbitMQ 三者的核心区别？如何选型？
标准答案：
维度KafkaRocketMQRabbitMQ
吞吐量极高（百万QPS）高（十万QPS）中等（万QPS）
消息模型发布订阅(Partition)Topic+Tag+GroupExchange+Queue

选型建议：大数据流处理→Kafka；电商交易场景(秒杀/下单)→RocketMQ；企业消息/微服务解耦→RabbitMQ。
Q2: 如何保证消息不丢失？（生产者→Broker→消费者 全链路）
标准答案：
1️⃣ 生产者端：发送确认机制。RocketMQ用SendStatus.SEND_OK校验，Kafka设置acks=all。失败重试3次+本地数据库日志兜底。
2️⃣ Broker端：同步刷盘(flushDiskType=SYNC_FLUSH)+主从复制。Kafka设置replication.factor≥3，min.insync.replicas≥2。
3️⃣ 消费者端：手动ACK，先处理业务再提交offset。RocketMQ用ConsumeConcurrentlyStatus.RECONSUME_LATER返回，Kafka用commitSync()。
Q3: 消息重复消费怎么办？如何保证幂等？
标准答案：
消息队列"至少一次"投递是正常的，幂等由业务层保证：
// 方案1：数据库唯一索引（最可靠）
INSERT INTO order_log (msg_id, order_id, ...) 
VALUES (#{msgId}, #{orderId}, ...) 
// msg_id 建唯一索引，重复插入自动失败

// 方案2：Redis SetNX（高并发场景）
String key = "msg:consume:" + msgId;
Boolean success = redis.setIfAbsent(key, "1", Duration.ofHours(24));
if (Boolean.TRUE.equals(success)) {
    // 处理业务
    processOrder(msg);
}
三选一策略：数据库唯一键(强一致)/Redis原子标记(高并发)/业务状态机(已处理的订单不再二次扣库存)。
Q4: 如何保证消息顺序消费？
标准答案：
RocketMQ：MessageQueueSelector将同一订单ID的消息路由到同一个Queue，消费者使用MessageListenerOrderly顺序消费。
Kafka：同一partition内天然有序。生产者用key.hashCode() % partitionNum确保同类消息进同一partition。
关键：不能有并行消费——顺序消费必须单线程，`MessageListenerConcurrently`→替换为MessageListenerOrderly。
Q5: 消息积压了怎么办？（生产现场排查指南）
标准答案：
1️⃣ 紧急扩容：临时增加消费者实例，扩大消费并行度(RocketMQ:consumeThreadMin/Max)。
2️⃣ 降级处理：跳过非核心消息(如日志/通知)，新建topic分流。
3️⃣ 批量迁移：如果积压量巨大(百万级)，写一个临时consumer把老消息批量迁到新topic，用20台机器并行消费。
4️⃣ 根本原因排查：消费代码是否阻塞(慢SQL/死锁/网络超时)。
预防：监控消费延迟、设置告警阈值、日常压测消费能力。
Q6: RocketMQ的事务消息原理？
标准答案：
RocketMQ事务消息分三步（半消息机制）：
// 第一步：发送半消息（消费者不可见）
TransactionMQProducer producer = new TransactionMQProducer("group");
producer.setTransactionListener(new TransactionListener() {
    // 第二步：执行本地事务
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            // 执行本地事务（如扣库存）
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }
    // 第三步：Broker回查（本地事务状态未知时）
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 去数据库查是否已完成
        return isDone ? COMMIT_MESSAGE : ROLLBACK_MESSAGE;
    }
});
核心："半消息"对消费者不可见→执行本地事务→成功则提交→失败则回滚→未知则回查。保证本地事务和消息发送的原子性。
Q7: Kafka的Partition和Consumer Group是什么关系？
标准答案：
一个Partition只能被同Group内一个consumer消费（保证顺序），但可以被不同Group消费（广播）。
核心约束：同一Consumer Group内，consumer数量 ≤ partition数量。多出来的consumer会空转。
动态重平衡(Rebalance)：consumer加入/退出时，Coordinator重新分配partition。注意：Rebalance期间整个Group停止消费(Stop-the-world)，设置session.timeout.ms和max.poll.interval.ms避免频繁Rebalance。
Q8: 死信队列(DLQ)是什么？有哪些使用场景？
标准答案：
死信队列是无法被正常消费的消息的"收容所"。
进入死信的条件：消息被消费失败N次(RocketMQ默认16次)、消息过期(TTL)、队列满了。
场景： - 格式错误的消息(JSON解析失败)→DLQ→人工修复后重发 - 业务校验失败(余额不足)→DLQ→定时扫描补偿 - 下游系统宕机→DLQ→等恢复后批量重推
处理流程：消费失败→重试N次→进DLQ→监控告警→人工排查→修复后重投。
Q9: 你在秒杀项目里怎么用消息队列削峰的？
标准答案：
Redis Lua扣减库存成功→异步发RocketMQ消息→订单消费者匀速消费→落库。
// 秒杀Service核心逻辑
Long result = redisTemplate.execute(stockLuaScript, keys, ...);
if (result == 1) {  // 扣减成功
    OrderMessage msg = new OrderMessage(userId, productId, requestId);
    rocketMQTemplate.asyncSend("seckill-order-topic", msg, 
        new SendCallback() {
            public void onSuccess(SendResult r) { /* 发成功 */ }
            public void onException(Throwable e) { 
                // 发失败：保存到本地消息表 + 定时任务重试
                saveLocalMessage(msg);
            }
        });
}
削峰原理：秒杀QPS 5000→Redis扛住→RocketMQ缓冲→消费者按1500/秒匀速处理→MySQL不会被打垮。
