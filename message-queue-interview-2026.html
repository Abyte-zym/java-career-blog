Q1：消息队列解决了什么问题？为什么不用直接调接口？

标准答案：消息队列主要解决三个问题——削峰填谷解耦异步处理。直接调接口在高并发下会导致系统雪崩；MQ把请求暂存队列，消费者匀速处理，保护下游。

👉 简历亮点：我的秒杀系统中，秒杀成功的请求先发到RocketMQ，订单服务异步消费，下游压力降低70%。

Q2：RocketMQ和Kafka怎么选？

标准答案：

维度RocketMQKafka
定位业务消息日志/流计算
事务消息✅ 原生支持⚠️ 需额外实现
延迟消息✅ 18个级别❌ 不支持
顺序消息✅ 队列级FIFO✅ 分区内有序
吞吐量10万级/秒百万级/秒

"秒杀场景选RocketMQ,因为需要事务消息保证下单和库存扣减一致性;日志采集选Kafka,因为吞吐量更高。"

Q3：如何保证消息不丢失？

标准答案：三端保障——

① 生产者端：使用同步发送+重试机制，send(msg, callback) 确认Broker收到

② Broker端：同步刷盘（flushDiskType=SYNC_FLUSH）+主从同步复制

③ 消费者端：手动ACK（consumeMessage返回CONSUME_SUCCESS），处理完再确认

// 生产者同步发送
SendResult result = producer.send(msg);
if (result.getSendStatus() != SEND_OK) {
    // 重试或落库补偿
    retryOrSave(msg);
}

Q4：消息重复消费怎么处理？

标准答案：MQ保证至少一次投递，但可能重复。消费者做幂等处理——唯一键去重Redis setNX数据库唯一索引。

// 消费端幂等：用消息ID+业务键去重
String dedupKey = "msg:" + msg.getMsgId();
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent(dedupKey, "1", 24, TimeUnit.HOURS);
if (!success) return CONSUME_SUCCESS; // 已处理，跳过

Q5：如何保证消息顺序消费？

标准答案：RocketMQ中，同一笔订单的消息发到同一个MessageQueue（通过订单ID取模），消费者端用MessageListenerOrderly单线程消费该队列，保证FIFO。

// 顺序发送：相同订单号→同一队列
msg.setKeys(orderId);
SendResult result = producer.send(msg, 
    (mqs, m, arg) -> {
        int idx = Math.abs(arg.hashCode()) % mqs.size();
        return mqs.get(idx);
    }, orderId);

Q6：什么是事务消息？秒杀场景怎么用？

标准答案：RocketMQ事务消息分三步——①发送半消息(half msg) ②执行本地事务(扣库存) ③根据本地事务结果提交或回滚。如果④步骤超时, Broker会回查本地事务状态。

👉 秒杀中：半消息→Redis Lua扣库存→成功则提交消息→订单服务消费落库。失败则回滚。

Q7：死信队列是什么？什么情况会进入死信？

标准答案：消息消费失败达到最大重试次数(默认16次)后，会被投递到死信队列(DLQ)。人工介入处理补偿。常见原因：业务逻辑bug、下游服务宕机、数据格式错误。

Q8：消息堆积怎么排查和解决？

标准答案：三步排查——监控Dashboard看消费延迟 → 检查消费者日志是否有异常 → 扩容消费者实例或临时提升消费线程数。

紧急方案：新建一个临时Topic，把积压消息转发过去，加10倍消费者快速消费。

Q9：消息队列的push和pull模式有什么区别？

标准答案：

Push模式：Broker主动推给消费者，实时性好，但可能压垮消费者。

Pull模式：消费者主动拉取，自主控制速率，但可能有延迟。

RocketMQ默认是长轮询(Long Polling)——消费者发起pull请求，Broker有消息立即返回，没消息挂起15秒。

Q10：你项目中RocketMQ是怎么用的？

标准答案："秒杀项目中，用户秒杀请求到达后，Redis Lua脚本扣减库存成功→发送下单消息到RocketMQ→订单消费服务异步落库。我用消息表+定时对账保证最终一致性，即使MQ宕机也不会丢订单。"
