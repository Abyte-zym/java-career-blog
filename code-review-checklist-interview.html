面试官考察点：代码审查(CodReview)是区分"只会写代码"和"有工程素养"的关键。答好这10题，实习生面试直接上分。

Q1：代码审查(Code Review)的核心目标是什么？不是找茬！

三大核心目标：
1. 知识共享：让团队成员了解彼此的代码，避免"只有一个人懂"的单点风险
2. 代码质量：发现逻辑错误、性能问题、安全隐患，在合并前拦截
3. 规范统一：保持代码风格和架构一致性，降低维护成本

面试加分：提一句"Code Review不是找bug——测试应该找bug，Review是找更优解"。

Q2：一份Java代码审查Checklist包含哪些要点？

10项必查清单：
1. ✅ 空指针防御：方法入参是否判空？Optional是否滥用？
2. ✅ 异常处理：是否吞异常？是否用了e.printStackTrace()？
3. ✅ 资源关闭：IO流/数据库连接是否在finally或try-with-resources中关闭？
4. ✅ 并发安全：共享变量是否线程安全？synchronized范围是否过大？
5. ✅ SQL注入：是否用了#{}而非${}？动态SQL是否有白名单？
6. ✅ 性能：循环里是否有DB查询？是否该用批量操作？
7. ✅ 日志规范：是否有关键操作日志？异常日志是否带堆栈？
8. ✅ 命名规范：变量名是否表达意图？方法名是否符合团队规范？
9. ✅ 单元测试：是否有测试覆盖边界条件？覆盖率是否达标？
10. ✅ 文档注释：复杂逻辑是否有注释说明为什么这么做？

Q3：代码审查中如何发现空指针问题？

审查时重点看：
// ❌ 危险：返回null
public User findById(Long id) {
    return userMap.get(id);  // 可能返回null，调用方不知道
}

// ✅ 安全：用Optional明确表达
public Optional<User> findById(Long id) {
    return Optional.ofNullable(userMap.get(id));
}

// ✅ 安全：方法内部防御
public void process(User user) {
    Objects.requireNonNull(user, "user不能为null");
    // 业务逻辑
}
三项必查：
- 集合操作是否判空（Map.get()、List.stream()）
- 链式调用是否有null风险（A.getB().getC()）
- equals()是否把常量放前面（"abc".equals(str)而非str.equals("abc")）

Q4：代码审查中常见的资源泄露场景有哪些？

五种常见泄露：
// ❌ 传统IO可能泄露
InputStream is = new FileInputStream("file.txt");
// 如果中间抛异常，is永远不会关闭

// ✅ try-with-resources自动关闭
try (InputStream is = new FileInputStream("file.txt")) {
    // 自动关闭，无需finally
}

// ❌ 数据库连接泄露
Connection conn = dataSource.getConnection();
// 忘记归还连接池

// ❌ Redis连接泄露
Jedis jedis = pool.getResource();
// 忘记 jedis.close()

// ❌ 线程池未关闭
ExecutorService executor = Executors.newFixedThreadPool(10);
// 忘记 executor.shutdown()
审查要点：所有实现了AutoCloseable的对象，是否在try-with-resources中或者在finally中关闭。

Q5：代码审查中如何发现并发安全问题？

四类并发问题审查：
1. 共享变量：Service层的成员变量是否线程安全？是否该用ThreadLocal？
2. 集合选型：ArrayList是否被多线程访问？该换CopyOnWriteArrayList还是Collections.synchronizedList？
3. 锁范围：synchronized块是否锁了不该锁的对象？是否可能死锁？
4. 原子操作：i++、check-then-act是否该用AtomicInteger或ConcurrentHashMap

// ❌ 危险：i++不是原子操作
private int counter = 0;
public void increment() { counter++; }

// ✅ 改用AtomicInteger
private AtomicInteger counter = new AtomicInteger(0);
public void increment() { counter.incrementAndGet(); }

Q6：代码审查时如何评估异常处理是否合理？

四个评判标准：
// ❌ 吞异常——最严重的错误
try { doSomething(); } catch (Exception e) {}

// ❌ 只打日志不处理
try { doSomething(); } catch (Exception e) {
    e.printStackTrace();  // 生产环境丢失
}

// ✅ 正确的分级处理
try {
    doSomething();
} catch (BusinessException e) {
    log.warn("业务异常: {}", e.getMessage());
    return R.error(e.getCode(), e.getMessage());
} catch (Exception e) {
    log.error("系统异常", e);
    return R.error(500, "系统繁忙");
}
审查清单：是否分类处理异常？是否记录完整堆栈？是否在finally中清理资源？是否有兜底响应保证用户体验？

Q7：代码审查中SQL相关问题的审查要点？

七大SQL审查要点：
1. MyBatis #{} vs ${}：动态排序字段必须白名单校验，禁止直接${orderBy}
2. N+1查询：循环里是否有数据库操作？该用批量查询或JOIN
3. 索引使用：WHERE条件是否符合最左前缀？是否该加EXPLAIN
4. SELECT *：是否只查需要的字段？大字段是否单独查询
5. 大批量操作：是否用了分批处理而非一次性更新百万行
6. 事务范围：@Transactional是否包了不该包的慢操作（RPC/发消息）
7. 死锁风险：多表更新顺序是否一致？是否使用了索引避免间隙锁

Q8：代码审查中发现重复代码怎么处理？

三原则：
1. 三次法则(Rule of Three)：相同逻辑出现第三次时，必须抽公共方法
2. 抽象层次一致：抽出来的公共方法要在同一抽象层级，不要上帝方法
3. 不要过度抽象：两处相似但语义不同的代码，不要为了消除重复而强行合并

// ❌ 重复代码
public void createUser() {
    log.info("操作开始");
    // 业务逻辑...
    log.info("操作结束");
}
public void updateUser() {
    log.info("操作开始");
    // 业务逻辑...
    log.info("操作结束");
}

// ✅ 用AOP或模板方法消除重复——这正是你Seckill项目中@SentinelResource的思想

Q9：SonarQube/阿里巴巴代码规约在代码审查中怎么用？

分级审查策略：
- Blocker（阻断）：必须修。如SQL注入、空指针、资源泄露
- Critical（严重）：强烈建议修。如性能问题、潜在的并发问题
- Major（主要）：有条件的修。如复杂度过高、代码重复
- Minor（次要）：可延后。如命名规范、注释不完整

面试加分："我们团队使用阿里的P3C插件+SonarQube做静态扫描，配合人工Code Review。静态工具扫语法，人工审逻辑。"

Q10：作为实习生，你的代码被Review时应该怎么做？

正确的态度（面试官最看重这个）：
1. 不要玻璃心：Review是对事不对人，批评代码不是批评你
2. 主动提问：不理解的意见及时问，不要自己猜
3. 记录下来：同样的错误不犯第三次。做一个"我踩过的坑"笔记
4. 下次先自查：提交前自己对着Checklist过一遍

黄金回答："作为实习生，我把Code Review当成学习机会。每次被指出问题，我都记下来，下次提PR前先自查。三个月下来，Review评论从十几条降到一两条。"
