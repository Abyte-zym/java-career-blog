❓ 1. 如何设计一个短URL系统（TinyURL）？
核心方案：分布式ID生成 + Base62编码 + 302重定向。
详细流程：① 接收长URL → ② 用雪花算法生成唯一ID → ③ ID转Base62（0-9a-zA-Z）得短码 → ④ 存MySQL（短码-长URL映射）+ Redis缓存 → ⑤ 用户访问短链时302跳转。
关键考量：QPS预估、短码长度（7位Base62支持62⁷≈3.5万亿）、过期策略、防止重复生成（布隆过滤器快速判断URL是否存在）。
Base62雪花算法302跳转布隆过滤器
❓ 2. 如何设计一个分布式限流器？
三层架构：网关层（Nginx/Sentinel）+ 服务层（Redis令牌桶）+ 业务层（信号量）。
算法选型：
  • 固定窗口：简单但有突刺问题（边界处双倍流量）
  • 滑动窗口：Redis ZSet实现，`zremrangeByScore`清理过期 + `zcard`计数，精准但内存开销大
  • 令牌桶（推荐）：Redis Lua脚本原子操作，`local tokens = min(capacity, lastTokens + rate * (now - lastTime)); if tokens >= 1 then lastTokens = tokens - 1; return 1; else return 0; end`
  • 漏桶：恒定速率处理，适合保护下游
滑动窗口令牌桶Redis LuaSentinel
❓ 3. 如何设计一个实时排行榜系统？
技术选型：Redis Sorted Set（ZSet）天生适合排行榜。
实现方案：① 积分变动时 `ZINCRBY leaderboard score userId` → ② 查Top100 `ZREVRANGE leaderboard 0 99 WITHSCORES` → ③ 查用户排名 `ZREVRANK leaderboard userId` → ④ 定时任务每小时归档到MySQL。
进阶优化：分区排行榜（日榜/周榜/总榜）→ 用不同Key区分。千万用户时用分桶策略（按userId hash分散到多个ZSet，查询时合并Top K）。
Redis ZSetZREVRANGE分桶策略
❓ 4. 如何设计一个Feed流系统（朋友圈/微博）？
核心模式：推模式 vs 拉模式 vs 推拉结合。
推模式（写扩散）：发动态时写入所有粉丝的收件箱（Redis List），适合小V。
拉模式（读扩散）：查询时从关注者列表拉取动态，适合大V。
推拉结合（推荐）：普通用户推模式，大V拉模式。大V的Feed单独存储，查询时合并。
技术实现：Redis List存储Feed ID + MySQL存储Feed详情 + 时间线生成定期快照。
推拉结合Feed流Redis List
❓ 5. 如何设计一个秒杀系统？
整体架构：前端限流 → 网关鉴权 → Redis原子扣减 → MQ异步下单 → DB落库。
核心要点：
  • 防超卖：Redis Lua脚本原子性扣减库存（检查→扣减→记录用户）
  • 防重复下单：用户ID + 商品ID做Key，setIfAbsent
  • 削峰：RocketMQ消息队列异步处理，订单服务慢慢消费
  • 限流：Sentinel QPS限流 + 前端按钮防抖
  • 缓存预热：秒杀前把库存预加载到Redis

❓ 7. 消息队列选型：Kafka vs RocketMQ vs RabbitMQ？
快速决策：
  • Kafka：超高吞吐（百万级QPS），适合日志/埋点/流计算，不支持延迟消息和事务消息（新版本支持但复杂）
  • RocketMQ：阿里出品，支持事务消息+延迟消息+顺序消息，适合电商交易场景
  • RabbitMQ：Erlang编写，低延迟，管理界面好，适合企业内部微服务通信，吞吐量中等
KafkaRocketMQRabbitMQ事务消息
❓ 8. 如何设计缓存架构？缓存一致性怎么保证？
缓存模式：Cache Aside（旁路缓存）最常用 → 读：先查缓存，miss查DB并回写；写：先更新DB再删缓存。
一致性方案：① 先更新DB后删除缓存（推荐）② 延迟双删（删缓存→更新DB→延迟再删一次）③ 订阅MySQL binlog（Canal）异步更新缓存 ④ 设置合理过期时间兜底。
多级缓存：本地缓存(Caffeine) → Redis → DB。本地缓存用短TTL(5-10s)降低不一致风险。
Cache Aside延迟双删Canal多级缓存
❓ 9. 如何设计一个搜索引擎？
核心组件：爬虫 → 分词器 → 倒排索引 → 检索 → 排序。
技术选型：Elasticsearch（基于Lucene），内置倒排索引+分布式+近实时搜索。
倒排索引原理：词→文档ID列表。例如"Java"→[doc1,doc3,doc7]，比正排索引（文档→词列表）检索快N倍。
实践要点：分词器选择（IK）、同义词处理、相关性打分（TF-IDF/BM25）、分片与副本。
Elasticsearch倒排索引TF-IDFLucene
❓ 10. 如何设计一个文件存储系统（图床/OSS）？
架构分层：接入层(鉴权+限流) → 业务层(元数据管理) → 存储层(对象存储)。
核心设计：① 文件上传：前端直传OSS → 回调通知服务器 → 记录元数据 ② 存储方案：MinIO（自建）/阿里云OSS（云服务）③ CDN加速 ④ 图片处理：缩略图/水印/格式转换。
扩展考量：分片上传大文件、断点续传、去重（MD5校验）、防盗链（签名URL+Referer）。
对象存储CDN分片上传MinIO
