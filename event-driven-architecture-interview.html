Q1：什么是事件驱动架构？和传统的请求-响应模式有什么区别？

事件驱动架构（EDA）：服务之间通过"事件"通信，而不是直接调用。生产者发布事件→事件总线→消费者订阅处理。

对比传统RPC/HTTP：
| 请求-响应 | 事件驱动 |
| 同步等待 | 异步解耦 |
| 调用方必须知道被调用方 | 生产者不关心谁消费 |
| 强依赖，一个挂了全链崩溃 | 松耦合，可独立部署 |
| 适合读多写少的查询 | 适合异步处理的写操作 |

面试时说："事件驱动就是把'你去做XX'变成'XX发生了'。比如下单后不是直接调库存服务减库存，而是发一个OrderCreated事件，库存服务自己订阅处理。"

Q2：Spring Event怎么用？@EventListener和@TransactionalEventListener有什么区别？

基本用法： 
// 1. 定义事件
public class OrderCreatedEvent {
    private Long orderId;
}
// 2. 发布事件
@Autowired private ApplicationEventPublisher publisher;
publisher.publishEvent(new OrderCreatedEvent(orderId));
// 3. 监听事件
@EventListener
public void handleOrderCreated(OrderCreatedEvent event) {
    // 处理逻辑
}
关键区别：
- @EventListener：事件一发出就执行，不管事务是否提交。如果事件处理失败不影响事务。
- @TransactionalEventListener(phase=AFTER_COMMIT)：事务提交后才执行。如果事务回滚，事件不处理。

面试要点：秒杀项目中，订单创建后发短信通知应该用AFTER_COMMIT——只有订单真写库了才通知用户。

Q3：CQRS是什么？什么时候该用它？

CQRS（Command Query Responsibility Segregation）：将读操作和写操作分离到不同模型。

核心思想： 
写模型(Command)：处理创建、更新、删除 —— 强一致性
读模型(Query)：   处理查询、列表、统计 —— 按需优化
什么时候用： - 读写比例严重失衡（100:1）→ 分别优化 - 查询需要多表JOIN → 读模型建宽表，一次查完 - 读模型可以存Redis/ES，写模型只写MySQL

秒杀项目例子：秒杀扣库存是Command（强一致），商品列表展示是Query（走缓存）。

Q4：Event Sourcing和传统CRUD有什么区别？

Event Sourcing（事件溯源）：不存当前状态，存所有改变状态的事件。当前状态=重放所有事件。

对比： 
传统CRUD：
UPDATE account SET balance=500 WHERE id=1;
→ 存结果，历史不可追溯

Event Sourcing：
INSERT INTO events VALUES ('存款100', '取款200', '存款600');
balance = 100-200+600 = 500
→ 存事件，可审计可回溯可时间旅行
审计场景：银行交易系统必须知道每笔钱怎么变的，Event Sourcing天然支持。

Q5：CDC（Change Data Capture）是什么？怎么实现？

CDC = 捕获数据库变更，转为事件流。

三种实现方式：
1. 触发器方式：在表上加trigger，变更时写事件表。侵入性强，性能差。
2. 日志解析（推荐）：读MySQL binlog / PostgreSQL WAL，解析出INSERT/UPDATE/DELETE。Debezium是主流开源方案。
3. 应用层双写：改数据库同时发消息。问题：可能不一致。

面试时说："秒杀项目中订单落库后，Debezium监听binlog自动发事件到Kafka，下游服务消费处理——零侵入、高性能。"

Q6：Saga模式怎么保证分布式事务？和Seata AT模式有什么区别？

Saga = 把一个长事务拆成多个本地事务+补偿。
每个步骤成功→执行下一步；失败→按顺序执行补偿操作（回滚）。

编排（Choreography）：每个服务自己知道下一步是谁。事件驱动，无中心协调器。
编排（Orchestration）：一个编排引擎（State Machine）负责调度。有中心协调器。

Saga vs Seata AT： | Saga | Seata AT |
| 最终一致性 | 强一致性 |
| 补偿模式 | 回滚模式 |
| 无中心/中心均可 | 必须有TC |
| 适合长事务 | 适合短事务 |
| 需要写补偿方法 | 自动回滚 |
面试时说："下单→扣库存→支付 这个流程用Saga，支付失败则取消订单→恢复库存。"

Q7：事件驱动架构中怎么保证消息顺序性和幂等性？

顺序性：
- Kafka：同一业务key（如orderId）发到同一partition，单partition内有序
- RocketMQ：顺序消息模式，同一队列内FIFO
- RabbitMQ：不支持全局顺序

幂等性：
1. 数据库唯一键：INSERT...ON DUPLICATE KEY UPDATE
2. Redis SETNX：setIfAbsent(事件ID, "处理中")，处理完删掉
3. 版本号：UPDATE SET balance=500, version=3 WHERE version=2
// 幂等消费示例
if (redis.setIfAbsent("event:" + eventId, "1", 24, TimeUnit.HOURS)) {
    process(event);  // 只有第一个消费者执行
} else {
    return;  // 重复消息直接丢弃
}

Q8：Spring Cloud Stream是什么？和直接用Kafka有什么区别？

Spring Cloud Stream = 消息中间件的抽象层。代码只依赖Stream API，底层切换Kafka→RabbitMQ不改业务代码。

核心概念： 
Supplier (生产者) → Binder (绑定器) → Channel (通道) → Consumer (消费者)
优势： - 中间件无关：换MQ只改pom和配置，不改代码 - 函数式编程：配合Spring Cloud Function，一个Function即消费者 - 内置重试+死信队列+分区支持

面试时说："秒杀项目的RocketMQ消费就是通过Stream封装的，如果将来切Kafka，改一行配置就行。"

Q9：事件驱动怎么写单元测试？怎么Mock ApplicationEventPublisher？

单元测试要点： 
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    @Mock ApplicationEventPublisher publisher;
    @InjectMocks OrderService orderService;
    
    @Test void shouldPublishEvent() {
        orderService.createOrder(orderDto);
        // 验证事件被发布
        verify(publisher).publishEvent(argThat(
            e -> ((OrderCreatedEvent)e).getOrderId().equals(1L)
        ));
    }
}
@EventListener测试：直接用Spring上下文或Mock方式：
- 单元测试：用Mockito验证EventListener方法被调用
- 集成测试：用@SpringBootTest + @TestEventPublisher

关键点：测试事件驱动时，验证"事件被发布了什么"比"事件被处理后发生了什么"更重要。

Q10：复盘一次事件驱动架构在生产中踩过的坑。

场景：订单创建后发积分事件，积分服务消费失败。

踩坑过程：
1. 订单创建→发OrderCreated事件
2. 积分服务消费→调用积分API超时→抛异常
3. 消息重试3次全失败→进入死信队列
4. 用户投诉"下单了没给积分"

解决方案：
1. 死信队列加监控告警（Prometheus + Grafana）
2. 消费失败写入补偿表（`event_compensation`）
3. 定时任务扫描补偿表，重试失败的积分发放
4. 幂等性保证：积分发放前查`point_log`表防重复

复盘总结："事件驱动不是银弹——必须有死信处理+补偿机制+监控告警三件套。只发事件不等于是最终一致性，得有兜底。"
