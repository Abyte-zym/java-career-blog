🔍 监控与可观测性面试10题

PrometheusGrafanaELKMicrometer链路追踪2026必考

监控可观测性是2026年Java后端面试新热点——大厂必问，中小厂常见。10道题帮你拿到面试分。

Q1什么是可观测性三大支柱？分别解决什么问题？
Metrics（指标）：回答"系统有没有问题"。如QPS、延迟P99、错误率。用Prometheus采集，Grafana展示。
Logging（日志）：回答"出了什么问题"。如错误堆栈、请求日志。用ELK（Elasticsearch+Logstash+Kibana）或Loki存储分析。
Tracing（链路追踪）：回答"问题出在哪个环节"。分布式系统中一个请求经过多个服务，用TraceId串联全链路。用Jaeger/Zipkin/SkyWalking实现。
💡 面试亮点：三者的关系——Metrics发现问题→Tracing定位节点→Logging查看细节。不是互相替代，是互补。
Q2Prometheus的核心架构是什么？为什么它比Zabbix更适合云原生？
Pull模型：Prometheus Server主动从Target拉取metrics（HTTP GET /metrics端点），而非被推送。
核心组件：Prometheus Server（采集+存储）→ Alertmanager（告警）→ Grafana（可视化）。
TSDB存储：自研时序数据库，按时间窗口压缩，适合海量指标存储。
PromQL：强大的查询语言，支持聚合、计算、函数。
服务发现：自动发现K8s Pod，不需要手动配置IP。
# Prometheus配置示例：抓取Spring Boot Actuator暴露的metrics scrape_configs: - job_name: 'spring-boot-app' metrics_path: '/actuator/prometheus' static_configs: - targets: ['localhost:8080']
Q3Spring Boot项目中如何暴露Metrics给Prometheus？用到了什么依赖？
引入Micrometer——Spring Boot 2.x+默认的metrics门面：
<dependency> <groupId>io.micrometer</groupId> <artifactId>micrometer-registry-prometheus</artifactId> </dependency> # application.yml management: endpoints: web: exposure: include: health,info,prometheus metrics: tags: application: ${spring.application.name}
Micrometer自动采集JVM指标（堆内存/GC/线程池）、HTTP请求指标（count/sum/max）、数据源指标等。访问 /actuator/prometheus 就能看到所有metrics。
Q4ELK（ElasticSearch+Logstash+Kibana）日志系统的数据流是怎样的？
Filebeat（轻量）→ 采集日志文件，发送到Logstash或直接到ES。
Logstash（重量处理）→ 解析、过滤、转换日志格式（grok正则提取字段）。
Elasticsearch → 存储+索引日志，支持全文搜索。
Kibana → 可视化查询、Dashboard、告警。
💡 面试时说：实际项目中我们通过Filebeat采集容器日志，Logstash用grok提取traceId/level/message等字段到ES，再通过Kibana的Discover快速定位错误。
Q5分布式链路追踪的原理是什么？TraceId和SpanId怎么传递？
核心概念：
• TraceId：整个请求链路的唯一标识，从入口生成，贯穿所有服务。
• SpanId：链路中每个操作（一个RPC调用、一次DB查询）的唯一标识。
• ParentSpanId：父操作的SpanId，形成调用树。

传递方式：HTTP Header透传。如Jaeger使用 uber-trace-id header：
// 网关生成TraceId，放入MDC String traceId = UUID.randomUUID().toString(); MDC.put("traceId", traceId); // Feign调用时通过Interceptor传递 request.header("X-Trace-Id", traceId);
Q6Grafana Dashboard怎么设计？常用的Panel类型有哪些？
常用Panel：
• Time series（时序图）：QPS、延迟P99、错误率趋势。
• Stat（状态卡片）：当前在线用户数、Pod运行数。
• Gauge（仪表盘）：CPU使用率、内存使用率。
• Table（表格）：Top N慢SQL、Top N大对象。
• Heatmap（热力图）：延迟分布。

设计原则：第一行放"红灯指标"（错误率/可用性）→ 第二行放"黄灯指标"（P99延迟）→ 第三行放"绿灯指标"（QPS/资源使用率）。从上到下优先级递减。
Q7什么是指标聚合？PromQL中rate()、irate()、increase()有什么区别？
• rate(counter[5m])：计算5分钟内每秒平均增长率。适合QPS统计。
• irate(counter[5m])：计算区间内最后两个点的瞬时增长率。适合捕捉突发流量。
• increase(counter[5m])：计算5分钟内的总增量。适合统计请求总数。
• histogram_quantile(0.99, rate(...))：计算P99延迟。

# 实用PromQL rate(http_server_requests_seconds_count[5m]) # 5分钟内平均QPS histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) # P99延迟 increase(http_server_requests_seconds_count[1h]) # 1小时内请求总数
Q8生产环境CPU飙高到100%，你怎么排查？（监控实操题）
第一步：确认问题范围——Grafana看CPU趋势图，确认是持续高还是瞬时尖峰。同时看QPS是否有突变。
第二步：定位进程/线程：
# Linux命令定位 top -Hp $(pgrep -f java) # 找到CPU最高的线程PID printf '%x\n' PID # 线程PID转16进制 jstack PID | grep 16进制 -A20 # 查看该线程的堆栈
第三步：Arthas分析（阿里开源，面试加分项）：
dashboard # 实时面板，看哪个线程吃CPU thread -n 3 # 显示CPU最高的3个线程及堆栈 trace Class method # 追踪方法调用耗时
第四步：常见原因——死循环、频繁Full GC（内存泄漏导致）、正则表达式回溯、Jackson序列化大对象。
Q9怎么设计一个告警系统？什么是告警风暴？怎么解决？
告警设计四要素：
1. 合理阈值：不要设得太敏感。P99延迟 > 500ms 持续5分钟才告警，不是瞬时。
2. 告警分级：P0（服务不可用→电话+企微）、P1（延迟升高→企微群）、P2（磁盘使用率→邮件）。
3. 告警抑制：如果机器A宕机了，机器A上的所有告警只发一条，不要刷屏。
4. 静默期：计划内的变更/发布期间关闭告警。

告警风暴：一个故障引发上百条告警。解决方案——告警聚合（Alertmanager的group_by）、告警依赖（A宕机则B的告警抑制）、值班轮转。
Q10SIEM、APM、NPM这些缩写分别指什么？在你的实习项目中用过哪些？
• APM（应用性能监控）：SkyWalking、Pinpoint、Elastic APM——监控应用内部的延迟、错误、吞吐。
• SIEM（安全信息和事件管理）：Splunk、ELK Security——分析安全日志，检测入侵。
• NPM（网络性能监控）：监控网络延迟、丢包率。
• RUM（真实用户监控）：前端性能，页面加载时间、白屏时间。

实习面试话术："我在RAG项目中集成了Micrometer+Prometheus做指标采集，docker-compose里加了Grafana做Dashboard。在秒杀项目中用MDC+Logback做链路追踪，每个请求带TraceId，方便排查问题。"
