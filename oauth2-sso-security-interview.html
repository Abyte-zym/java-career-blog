Q1：OAuth2有哪四种授权模式？各自适用什么场景？

四种模式（按安全性从高到低）：

1. 授权码模式（Authorization Code）：最安全、最常用。用户→授权服务器→拿code→后端用code换token。微信登录、GitHub登录都用这个。
2. 客户端模式（Client Credentials）：没有用户参与，服务端之间调用。比如后端调用云服务API。
3. 密码模式（Resource Owner Password）：用户直接把账号密码给客户端。已被OAuth2.1废弃——不安全。
4. 简化模式（Implicit）：前端直接拿token。也被OAuth2.1废弃，换成PKCE增强的授权码模式。

面试加分：提一句"OAuth2.1已废弃密码模式和简化模式，推荐PKCE+授权码"。

Q2：JWT和传统Session有什么区别？什么时候用哪个？

核心区别：
维度JWTSession
存储位置客户端（localStorage/Cookie）服务端（Redis/内存）
扩展性✅ 无状态，天然支持分布式❌ 需要Session共享
安全性⚠️ 签发后无法主动失效✅ 服务端可随时删除
性能✅ 无需查Redis⚠️ 每次请求查Redis
体积⚠️ 较大（通常1KB+）✅ 仅一个SessionId

选型原则：微服务+分布式→JWT；单体应用+需要踢人→Session；两者可混合——JWT做认证+Redis存黑名单做退出。

Q3：SSO单点登录是怎么实现的？画一下核心流程

CAS协议为核心（最经典的SSO方案）：

流程（4步）：
1. 用户访问 app1.com → 无登录态 → 重定向到 sso.com/login
2. sso.com 验证Cookie → 已登录 → 生成Service Ticket（ST） → 重定向回 app1.com/?ticket=ST-xxx
3. app1.com 拿到ST → 后端发请求到 sso.com 验证ST → 返回用户信息 → 建立本地Session
4. 用户访问 app2.com → 同样流程 → sso.com已有Cookie → 无需重新输密码

一句话：所有应用信任同一个认证中心（SSO Server），通过票据（Ticket）传递认证结果。

Q4：Spring Security过滤器链是怎么工作的？

核心过滤器链（按顺序）：
SecurityContextPersistenceFilter  ← 恢复SecurityContext
    ↓
UsernamePasswordAuthenticationFilter ← 处理表单登录
    ↓
BasicAuthenticationFilter         ← 处理HTTP Basic认证
    ↓
BearerTokenAuthenticationFilter    ← 处理JWT Bearer Token
    ↓
ExceptionTranslationFilter        ← 认证异常→重定向登录页
    ↓
FilterSecurityInterceptor         ← 最后一道：鉴权决策
面试时说："Spring Security基于Servlet过滤器链，`DelegatingFilterProxy`是入口，把请求交给Spring管理的FilterChainProxy。每个请求经过15+个过滤器，最后在FilterSecurityInterceptor判断有没有权限。"

Q5：RBAC权限模型是什么？怎么设计数据库表？

RBAC = Role-Based Access Control（基于角色的访问控制）

5张核心表：
用户表 user (id, username, password)
角色表 role (id, name, code)
权限表 permission (id, name, url, method)
用户角色关联表 user_role (user_id, role_id)
角色权限关联表 role_permission (role_id, permission_id)
查询用户权限： 
SELECT p.* FROM permission p
JOIN role_permission rp ON p.id = rp.permission_id
JOIN user_role ur ON rp.role_id = ur.role_id
WHERE ur.user_id = ?
面试加分：提一句"RBAC3 = RBAC0（基础）+ RBAC1（角色继承）+ RBAC2（约束分离）"。

Q6：access_token过期了怎么办？refresh_token机制怎么设计？

双Token机制：
access_token  - 15分钟过期，每次请求携带
refresh_token - 7天过期，只用来换新access_token
刷新流程：
1. 请求返回401 → 客户端自动调 /oauth/token (grant_type=refresh_token)
2. 服务端验证refresh_token未过期且未被吊销
3. 返回新的access_token + 新的refresh_token（可选）
4. 旧refresh_token作废（防止被盗用后无限刷新）

安全设计：refresh_token只传一次、用完即换、存在HttpOnly Cookie而非localStorage。

Q7：CSRF攻击怎么防御？Spring Security里怎么配置？

CSRF（跨站请求伪造）：攻击者诱导用户点击链接，利用用户已登录态发恶意请求。

防御方案：
1. CSRF Token（传统方案）：服务端生成随机token放在表单隐藏字段，提交时校验。
2. SameSite Cookie（现代方案）：Set-Cookie: SameSite=Strict——跨站请求不带Cookie。
3. 验证Referer/Origin头：检查请求来源是否合法。

Spring Security配置： 
http.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
    .ignoringRequestMatchers("/api/public/**")
);
注意：前后端分离+JWT的项目，CSRF威胁小（因为JWT不在Cookie里），可以 .csrf().disable()。

Q8：OAuth2的授权码模式为什么比密码模式安全？

核心区别：密码有没有经过前端。

密码模式（不安全）：用户把密码直接给客户端→客户端拿密码换token。风险：恶意客户端可以记录密码，用户不知道客户端在后台做了什么。

授权码模式（安全）：
1. 客户端重定向到授权服务器（如GitHub登录页）
2. 用户在授权服务器上输入密码——密码不经过客户端
3. 授权服务器返回授权码（一次性、短期有效）
4. 客户端后端用授权码+client_secret换token

为什么安全：密码只在授权服务器输入，客户端从未见过密码。即使客户端被攻破，攻击者拿不到用户密码，只能拿到有时效的token。

Q9：分布式Session有哪几种方案？优缺点是什么？

4种方案：

1. Session复制（Tomcat集群）：每台机器广播Session。❌ 网络开销大，机器多了不可行。
2. Session粘滞（IP Hash）：Nginx ip_hash保证同一用户打到同一台。❌ 机器挂了Session全丢。
3. 集中式Session（Redis）：所有机器读同一个Redis。✅ 推荐方案，只需统一序列化。
4. 无状态（JWT）：不用Session，每次请求带Token。✅ 微服务最佳实践。

你说的秒杀项目：用JWT做认证，Redis存黑名单（登出+密码修改时加入）。既有无状态的扩展性，又有服务端控制能力。

Q10：如何实现"我被踢下线"功能？

方案：Redis黑名单 + WebSocket推送

Session模式（简单）：
管理员删除对应Session → 用户下次请求→服务端检查Session不存在→返回登录页

JWT模式（复杂）：
JWT签发后无法主动失效→需要加一层Redis黑名单：
// 踢人时
redis.set("token:blocked:" + userId, currentTime, 剩余过期时间)

// 每次请求校验
if (redis.exists("token:blocked:" + userId) 
    && redis.get("token:blocked:" + userId) > token签发时间) {
    throw new UnauthorizedException("您已被强制下线");
}
配合WebSocket：踢人后发WebSocket消息→前端弹出"你已被管理员踢下线"→跳转登录页。
