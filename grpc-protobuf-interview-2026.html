Q1：gRPC和REST有什么区别？什么时候选gRPC？

核心区别：
| | gRPC | REST | |--|------|------| | 协议 | HTTP/2 | HTTP/1.1 | | 数据格式 | Protobuf(二进制) | JSON(文本) | | 接口定义 | .proto文件 | OpenAPI/Swagger | | 性能 | 高(二进制+多路复用) | 相对低 | | 浏览器支持 | 需grpc-web | 原生支持 | | 代码生成 | 自动生成客户端 | 手动/OpenAPI生成 | 选gRPC的场景：微服务间通信、低延迟要求、需要流式传输、多语言环境。
选REST的场景：对外开放API、需要浏览器直接调用、团队对Protobuf不熟。

Q2：Protobuf为什么比JSON快？编码原理简单说

3个原因：
1. 二进制编码：字段用数字标签(1,2,3)而非字符串key，数据体积小很多
2. Varint压缩：整数用变长编码，小数字只占1-2字节
3. 无Schema开销：发送方和接收方都有.proto定义，只传值不传结构

面试加分：提到Protobuf向后兼容——加了新字段，旧版能忽略，新版能读到默认值。因为用数字标签而非字段名。

Q3：gRPC四种服务模式是什么？各用在什么场景？

// 1. 一元RPC（Unary）—— 一问一答
rpc GetOrder(OrderRequest) returns (OrderResponse);

// 2. 服务端流式 —— 客户端发一次，服务端持续推送
rpc ListOrders(OrderFilter) returns (stream Order);

// 3. 客户端流式 —— 客户端持续发送，服务端汇总回复
rpc UploadLogs(stream LogEntry) returns (UploadResult);

// 4. 双向流式 —— 双方独立读写
rpc Chat(stream Message) returns (stream Message);
场景：
- 一元：普通API调用
- 服务端流：订阅推送、大文件下载
- 客户端流：批量上传、日志收集
- 双向流：聊天、实时协作

Q4：gRPC拦截器（Interceptor）怎么用？和Spring AOP什么关系？

两种拦截器：
客户端拦截器：发请求前自动加token/日志/超时
服务端拦截器：收请求时做鉴权/限流/监控

// 客户端拦截器示例
public class AuthInterceptor implements ClientInterceptor {
    public  ClientCall interceptCall(
        MethodDescriptor method, CallOptions callOptions, Channel next) {
        return new ForwardingClientCall.Simple(next.newCall(method,callOptions)) {
            public void start(Listener responseListener, Metadata headers) {
                headers.put(Metadata.Key.of("Authorization",
                    Metadata.ASCII_STRING_MARSHALLER), "Bearer xxx");
                super.start(responseListener, headers);
            }
        };
    }
}
和Spring AOP的对比：思想一致——不侵入业务代码做增强。gRPC拦截器绑定RPC生命周期，AOP绑定Bean生命周期。

Q5：HTTP/2的哪些特性支撑了gRPC的性能？

3个关键特性：
1. 多路复用（Multiplexing）：一个TCP连接承载多个并发请求，解决HTTP/1.1的队头阻塞
2. 头部压缩（HPACK）：请求头用静态表+动态表压缩，Protobuf元数据体积更小
3. 服务端推送（Server Push）：服务端可主动推数据（gRPC流式基于此）

面试时画个对比：HTTP/1.1是单车道一条龙排队，HTTP/2是多车道同时跑。

Q6：Protobuf的向后兼容性怎么保证？

核心规则：
- 不能改已有字段的编号（数字标签是身份的标识）
- 新增字段用新编号，旧客户端会自动忽略
- 删除字段时标记 reserved，防止编号被重用
- 不要改字段类型（int32改成string=灾难）

message User {
  reserved 2, 15, 9 to 11;  // 已废弃的字段号
  reserved "old_email", "nickname";  // 已废弃的字段名
  int64 id = 1;
  string name = 3;  // 原来email在2，已废弃；新字段用3
}

Q7：gRPC的Deadline/Timeout怎么设置？超时后行为？

设置方式： 
// 客户端设置1秒超时
stub.withDeadlineAfter(1, TimeUnit.SECONDS).getOrder(request);
超时后：
- 客户端收到 Status.DEADLINE_EXCEEDED
- 服务端可以通过 Context.current().getDeadline() 检查是否已超时，避免做无用功
- gRPC会自动取消正在进行的RPC

面试加分：提到 Deadline 是绝对时间，Timeout 是相对时长。Deadline会在调用链上传递——A调B时，B的Deadline=min(A的Deadline, B自己的Deadline)。

Q8：gRPC的负载均衡怎么实现？

两种模式：
1. 服务端LB（推荐）：Nginx/Envoy在前面做代理层负载均衡
2. 客户端LB：客户端通过NameResolver从注册中心拿全部实例，自己轮询

客户端LB示例（配合Nacos）： 
channel = ManagedChannelBuilder.forTarget("nacos://my-service")
    .defaultLoadBalancingPolicy("round_robin")
    .usePlaintext()
    .build();
面试时对比：微服务里gRPC通常配客户端LB+长连接——和Dubbo一样，省去一层网络跳转。

Q9：gRPC中如何处理错误？和HTTP状态码的对应关系？

gRPC用 Status 而非HTTP状态码：
| gRPC Status | HTTP Code | 含义 | |-------------|-----------|------| | OK | 200 | 成功 | | INVALID_ARGUMENT | 400 | 参数错误 | | UNAUTHENTICATED | 401 | 未认证 | | PERMISSION_DENIED | 403 | 无权限 | | NOT_FOUND | 404 | 资源不存在 | | INTERNAL | 500 | 内部错误 | | UNAVAILABLE | 503 | 服务不可用 | | DEADLINE_EXCEEDED | 504 | 超时 | 
// 服务端返回错误
responseObserver.onError(
    Status.NOT_FOUND.withDescription("订单不存在").asRuntimeException()
);
// 客户端处理
catch (StatusRuntimeException e) {
    if (e.getStatus().getCode() == Status.Code.NOT_FOUND) { ... }
}

Q10：.proto文件怎么定义？Service注解编译后生成什么？

完整.proto示例： 
syntax = "proto3";
package com.example.order;

service OrderService {
  rpc CreateOrder(CreateOrderReq) returns (CreateOrderResp);
  rpc GetOrder(GetOrderReq) returns (Order);
}

message CreateOrderReq {
  int64 user_id = 1;
  repeated OrderItem items = 2;  // repeated=数组
}
message OrderItem {
  int64 product_id = 1;
  int32 quantity = 2;
}
编译后生成：
- OrderServiceGrpc.java：包含 OrderServiceImplBase（服务端继承）、OrderServiceStub（客户端异步）、OrderServiceBlockingStub（客户端同步）
- CreateOrderReq.java：每个message生成一个类，含builder模式
