Q1：@Transactional 注解的核心原理是什么？

核心答案：AOP动态代理。Spring为加了@Transactional的Bean生成代理对象，在方法调用前后通过TransactionInterceptor拦截，执行开启事务→执行业务→提交/回滚的逻辑。

关键细节：
- JDK代理（接口）或CGLIB代理（类）
- 实际执行：PlatformTransactionManager → 调用DataSourceTransactionManager → 操作JDBC Connection
- 异常回滚：默认只对RuntimeException和Error回滚，受检异常不回滚！

Q2：事务传播行为7种分别是什么？常用哪几种？

7种传播行为：
传播行为说明
REQUIRED（默认）有事务则加入，没有则新建
REQUIRES_NEW始终新建独立事务，挂起当前事务
SUPPORTS有事务就加入，没有就非事务执行
NOT_SUPPORTED非事务执行，挂起当前事务
MANDATORY必须在事务中，否则抛异常
NEVER必须非事务，否则抛异常
NESTED嵌套事务，内层回滚不影响外层（需要savepoint支持）

最常用：REQUIRED（默认，80%场景）和 REQUIRES_NEW（独立事务，如日志记录不受主事务回滚影响）。

Q3：事务隔离级别有哪些？各解决什么问题？

4种隔离级别（从低到高）：
隔离级别脏读不可重复读幻读
READ_UNCOMMITTED❌❌❌
READ_COMMITTED✅❌❌
REPEATABLE_READ（MySQL默认）✅✅⚠️部分解决
SERIALIZABLE✅✅✅

面试加分：MySQL InnoDB的REPEATABLE_READ通过MVCC+间隙锁解决了幻读。Oracle默认READ_COMMITTED。

Q4：@Transactional 在哪些场景下会失效？

6大失效场景（面试高频）：
1. 同类内部方法调用：this.methodB() 不走代理，事务不生效 → 用@Autowired注入自己或AopContext.currentProxy()
2. 方法非public：Spring AOP只拦截public方法（CGLIB可拦截protected）
3. 异常类型不匹配：默认只回滚RuntimeException，受检异常要加rollbackFor = Exception.class
4. 异常被catch吞掉：try-catch后不重新抛出，事务正常提交
5. 数据库引擎不支持：MyISAM不支持事务，@Transactional无效
6. 多线程：事务绑定在线程上，新线程是新事务

// 最常见的坑：同类调用
@Service
public class UserService {
    @Transactional
    public void methodA() {
        this.methodB(); // ❌ 事务不生效！this绕过了代理
    }
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() { ... }
}

Q5：编程式事务和声明式事务有什么区别？什么时候用编程式？

声明式事务（@Transactional）：注解驱动，Spring自动管理。适合90%场景。
编程式事务（TransactionTemplate）：代码手动控制事务边界。

编程式事务适用场景： 
transactionTemplate.execute(status -> {
    // 只回滚部分操作
    try { doA(); } catch (Exception e) { log.error("A失败"); }
    try { doB(); } catch (Exception e) { status.setRollbackOnly(); }
    return null;
});
什么时候用编程式：需要精细控制事务边界、部分失败不回滚整体、嵌套事务逻辑复杂时。

Q6：Spring事务和数据库事务是什么关系？

一句话：Spring事务是对数据库事务的封装，底层还是JDBC Connection的commit/rollback。

执行链路：
@Transactional → Spring AOP → TransactionInterceptor → DataSourceTransactionManager → connection.setAutoCommit(false) → 执行业务SQL → connection.commit() 或 rollback()

关键理解：事务最终由数据库执行，Spring负责管理事务的开始、提交和回滚时机。所以MyISAM引擎上@Transactional完全无效——因为数据库层面不支持。

Q7：分布式事务怎么处理？Seata AT模式原理是什么？

方案对比：
- 2PC（两阶段提交）：协调者→所有参与者prepare→全部ok才commit（性能差）
- TCC（Try-Confirm-Cancel）：业务层实现三阶段，侵入性强但性能好
- Seata AT：自动拦截SQL，生成before-image和after-image，回滚时反向补偿
- 消息最终一致性：用MQ+RocketMQ事务消息+本地消息表（最轻量）

Seata AT原理（面试用）： 
1. 一阶段：执行业务SQL + 记录undo_log（before-image+after-image）
2. 二阶段-提交：异步删除undo_log
3. 二阶段-回滚：根据undo_log反写数据
实习岗说这句就够了："秒杀项目用的是消息队列+本地消息表实现最终一致性，比Seata更轻量。"

Q8：@Transactional(readOnly = true) 有什么用？

三个作用：
1. MySQL优化：MySQL会对只读事务做优化（不维护事务ID等）
2. Spring优化：跳过flush，Hibernate脏检查关闭
3. 语义明确：代码意图清晰，读操作不加@Transactional(默认读写)可能被误解

最佳实践：查询方法加@Transactional(readOnly = true)，写方法不要加。

Q9：Spring事务超时怎么配置？超时后会发生什么？

配置： 
@Transactional(timeout = 30)  // 30秒超时
超时后：Spring会setRollbackOnly()并抛出TransactionTimedOutException。但注意：数据库端的SQL可能仍在执行——这是MySQL层面的限制，Spring无法强制kill。

面试时提一句：可以配合数据库层面的max_execution_time（MySQL 5.7+）或statement_timeout（PostgreSQL）做双重保护。

Q10：Spring事务和JPA/Hibernate事务有什么区别？

关系：Spring事务是抽象层，JPA/Hibernate事务是具体实现。Spring通过JpaTransactionManager桥接两者。

关键区别：
- Spring事务管理的是JDBC Connection级别
- JPA的EntityManager有自己的持久化上下文（一级缓存）
- Hibernate的flush时机受Spring事务影响：事务提交前自动flush
- JPA的乐观锁（@Version）在事务提交时检查

一句话总结：Spring管"什么时候提交"，JPA管"提交什么数据"。两者配合工作。
