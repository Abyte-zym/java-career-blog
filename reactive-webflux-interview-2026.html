🔥 为什么2026年要学Reactive？

趋势判断：Spring 6.x / Spring Boot 3.x 已将 WebFlux 作为一等公民。字节跳动的基础架构、阿里的高并发网关、美团的实时数据处理——都在用响应式编程。2026年面试，能讲清楚 Mono/Flux 是明显的差异化优势。

Q1: 什么是响应式编程？与传统命令式编程有什么区别？

核心答案：响应式编程是异步、非阻塞、事件驱动的编程范式。数据作为"流"（Stream）推送，系统对数据变化做出反应，而不是主动轮询。

对比：

维度命令式响应式
执行模型同步阻塞（一个线程等结果）异步非阻塞（回调/事件驱动）
线程模型Thread-per-Request（每个请求一个线程）Event Loop + 少量线程
适用场景低并发、CRUD业务高并发、IO密集、实时数据流
资源利用线程阻塞时浪费CPU少量线程服务大量请求

Q2: Mono 和 Flux 的区别？什么时候用哪个？

核心答案：

  • Mono<T>：返回 0或1个 元素。比如查询单个用户、保存后返回ID。
  • Flux<T>：返回 0到N个 元素。比如查询列表、实时消息流、WebSocket推送。
// Mono — 最多一个结果
Mono<User> user = userRepository.findById(1L);

// Flux — 多个结果
Flux<User> users = userRepository.findAll();

// 互相转换
Flux<User> flux = mono.flux();  // Mono → Flux
Mono<List<User>> mono = flux.collectList();  // Flux → Mono

面试金句："Mono 对标 Optional，Flux 对标 Stream。核心区别是数据元素个数——0-1用Mono，0-N用Flux。"

Q3: 什么是背压（Backpressure）？WebFlux 怎么处理？

核心答案：背压是指生产者速度 > 消费者速度时，消费者告诉生产者"慢一点"的机制。防止内存溢出。

WebFlux 的处理方式：

  • request(n)：消费者声明自己能处理n个元素
  • onBackpressureBuffer()：缓存多余数据
  • onBackpressureDrop()：丢弃多余数据
  • onBackpressureLatest()：只保留最新数据
  • limitRate()：限制生产速率
// 示例：限制速率 + 缓存策略
Flux.interval(Duration.ofMillis(1))    // 每1ms产生一个数据（快）
    .onBackpressureBuffer(100)          // 缓存100个
    .limitRate(50)                      // 每秒最多50个
    .subscribe(data -> process(data));  // 慢消费者

面试金句："背压是响应式系统的核心——Reactive Streams 规范通过 request(n) 实现从下游到上游的流量控制，这是传统 Future/CompletableFuture 做不到的。"

Q4: Spring MVC（Tomcat）vs Spring WebFlux（Netty）怎么选？

核心答案：

维度Spring MVC（Servlet）Spring WebFlux（Reactive）
运行时Tomcat/JettyNetty/Undertow
并发模型线程池（200线程≈200并发）Event Loop（少量线程服务万级并发）
数据库JDBC/JPA（阻塞）R2DBC（非阻塞）
学习曲线低高
适用场景传统CRUD、业务逻辑复杂高并发网关、实时推送、API聚合

选型原则：如果整个链路（Controller→Service→DAO）都是非阻塞的，用 WebFlux；如果有JDBC等阻塞组件，就用MVC。混合使用反而降低性能。

Q5: Reactor 中 map 和 flatMap 的区别？

核心答案：

  • map：同步转换，T → V。返回普通值。
  • flatMap：异步转换，T → Publisher<V>。返回 Mono/Flux，用于调用另一个异步方法。
// map：同步转换
Mono<String> name = userMono.map(User::getName);

// flatMap：异步转换（比如查完用户再查订单）
Mono<List<Order>> orders = userMono
    .flatMap(user -> orderService.findByUserId(user.getId()));

// 如果用 map 返回 Mono<Mono<Order>> — 嵌套了，不对

面试金句："map 解决同步转换，flatMap 解决异步编排——当你需要在响应式流里调用另一个返回 Publisher 的方法时，必须用 flatMap 拍平。"

Q6: WebFlux 的函数式端点（RouterFunction）和注解式（@RestController）怎么选？

核心答案：

  • @RestController：声明式，适合大多数场景，团队熟悉。
  • RouterFunction：函数式，适合需要动态路由、统一处理、函数组合的场景。
// 函数式风格
@Bean
public RouterFunction<ServerResponse> route(UserHandler handler) {
    return RouterFunctions
        .route(GET("/users/{id}"), handler::getUser)
        .andRoute(POST("/users"), handler::createUser)
        .andRoute(GET("/users"), handler::listUsers);
}

面试金句："RouterFunction 的优势是路由即数据——可以对路由做 filter、transform 等函数式操作。但团队协作首选 @RestController。"

Q7: Reactor 的 Schedulers 怎么设置？什么是执行模型？

核心答案：

Scheduler线程池适用场景
Schedulers.immediate()当前线程测试/简单场景
Schedulers.single()单线程复用顺序严格的状态操作
Schedulers.parallel()CPU核心数个线程CPU密集计算
Schedulers.boundedElastic()动态扩缩、有上限IO密集、网络调用（默认）
Flux.range(1, 100)
    .publishOn(Schedulers.parallel())  // 在并行线程执行
    .map(this::cpuIntensiveWork)       // CPU密集
    .subscribeOn(Schedulers.boundedElastic());  // 订阅在弹性线程

publishOn vs subscribeOn：subscribeOn影响上游订阅线程；publishOn影响下游操作线程。

Q8: R2DBC 是什么？和 JDBC 有什么区别？

核心答案：R2DBC（Reactive Relational Database Connectivity）是响应式关系型数据库驱动。JDBC是同步阻塞的，R2DBC是全异步非阻塞的。

对比：

// JDBC（阻塞）
User user = jdbcTemplate.queryForObject("SELECT * FROM users WHERE id=?", User.class, 1L);
// 线程阻塞等待数据库返回

// R2DBC（非阻塞）
Mono<User> user = databaseClient.sql("SELECT * FROM users WHERE id=$1")
    .bind(0, 1L)
    .map(row -> new User(row.get("id"), row.get("name")))
    .one();
// 立即返回 Mono，不阻塞线程

注意事项：R2DBC 不支持 JPA/Hibernate 的关联映射、懒加载等特性。适合简单CRUD+高并发场景。

Q9: 在 WebFlux 中怎么做异常处理？

核心答案：三种方式：

// 1. onErrorReturn — 返回默认值
userMono.onErrorReturn(new User("unknown"));

// 2. onErrorResume — 切换到备用流
userMono.onErrorResume(e -> {
    log.error("查询失败", e);
    return Mono.just(cachedUser);
});

// 3. @ExceptionHandler — 全局异常处理
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(UserNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public Mono<ErrorResponse> handleNotFound(UserNotFoundException e) {
        return Mono.just(new ErrorResponse(404, e.getMessage()));
    }
}

面试金句："onErrorReturn 用于降级兜底，onErrorResume 用于故障转移，@ExceptionHandler 用于统一格式。三种方式组合使用确保系统韧性。"

Q10: 你用 WebFlux 做过什么项目？踩过什么坑？

回答模板（结合你的RAG项目扩展）：

"在我的RAG智能知识库系统中，我用 WebFlux 实现了流式输出——调用大模型API后，通过 Flux 逐Token推送到前端，实现了打字机效果。相较于传统阻塞方式，WebFlux 让服务只用少量线程就支撑了100+并发对话。"

踩过的坑：

  • ❌ 在 flatMap 里用了阻塞调用（JDBC）→ 导致 Event Loop 线程堵塞 → ✅ 改用 R2DBC 或 Schedulers.boundedElastic() 隔离
  • ❌ subscribe() 忘记加 onError → 异常被吞 → ✅ 加 .doOnError() 日志
  • ❌ 背压没配置 → 内存溢出 → ✅ 加 limitRate + onBackpressureBuffer
