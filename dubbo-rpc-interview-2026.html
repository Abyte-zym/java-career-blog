Q1：RPC是什么？和HTTP调用有什么区别？什么时候用RPC？

RPC（Remote Procedure Call）：像调用本地方法一样调用远程服务。核心是"屏蔽网络细节"。

对比HTTP：
维度RPC（Dubbo）HTTP（Spring Cloud）
协议自定义TCP协议（更轻量）HTTP/1.1 文本协议
性能高：二进制序列化+长连接较低：文本传输+短连接
服务治理内置：注册发现/负载均衡/监控需额外组件
跨语言较差（Java原生）天然支持
适用场景Java微服务内部调用对外API/跨语言
面试金句："RPC追求高性能和透明化——调用者不感知网络。HTTP追求标准化和通用性。实践中网关层用HTTP，服务间用Dubbo。"

Q2：Dubbo的一次RPC调用完整流程是怎样的？

全链路10步：
1. 客户端Proxy：通过动态代理拦截方法调用
2. 负载均衡：从注册中心获取Provider列表，选一个节点
3. 协议编码：将请求（接口名+方法名+参数）序列化为二进制
4. 网络传输：通过Netty长连接发到Provider
5. 协议解码：Provider反序列化请求
6. 服务定位：通过接口名找到对应的实现类
7. 反射调用：invoke实际方法
8. 结果编码：序列化返回值
9. 网络回传：Netty返回给Consumer
10. 结果解码：Consumer反序列化得到返回值
// 客户端代码（完全无感知远程调用）
DemoService demoService = ReferenceConfig.get(DemoService.class);
String result = demoService.sayHello("world"); // 和本地调用一模一样

Q3：Dubbo支持哪些注册中心？Nacos和Zookeeper怎么选？

Dubbo支持的5种注册中心：
注册中心CAP适用场景
ZookeeperCP强一致性要求，节点少（<100）
NacosAP+CP可切换推荐！支持配置中心+服务发现二合一
RedisAP简单场景，已有Redis集群
ConsulCP多数据中心，健康检查强
EtcdCPK8s生态，Go实现
选型建议：新项目优先Nacos——同时解决注册+配置两个需求，减少组件。Zookeeper是经典选型，稳定性经过大厂验证。

Q4：Dubbo有哪些负载均衡策略？

5种内置策略：
策略原理适用场景
Random（默认）加权随机，权重越大概率越高通用场景
RoundRobin加权轮询，按序分配机器配置均衡
LeastActive分给当前活跃请求数最少的长连接/耗时不均
ConsistentHash同一参数值路由到同一Provider有状态服务/缓存命中
ShortestResponse分给历史响应最快的（Dubbo 3.0+）自动感知性能
// 配置示例
@DubboReference(loadbalance = "leastactive")
private UserService userService;

Q5：Dubbo的SPI机制是什么？和Java SPI有什么区别？

SPI（Service Provider Interface）：一种扩展机制——不修改源码就能替换实现。

Dubbo SPI vs Java SPI：
维度Java SPIDubbo SPI
加载方式一次性加载所有实现按需加载（懒加载）
扩展点无命名每个实现有唯一Key
AOP不支持支持Wrapper包装增强
IOC不支持支持setter注入
面试金句："Dubbo的微内核+插件架构就是靠SPI实现的——协议、注册中心、负载均衡全是SPI扩展点。不改核心代码就能换序列化协议。"

Q6：Dubbo支持哪些序列化协议？性能怎么排？

6种序列化协议对比：
协议性能跨语言可读性
Hessian2（默认）中✅二进制
Fastjson2高✅JSON文本
Kryo极高❌二进制
Protobuf极高✅二进制，需.proto
JDK原生低❌二进制
Avro高✅二进制+Schema
性能排序：Protobuf ≈ Kryo > Fastjson2 > Hessian2 > Avro > JDK。追求极致用Protobuf，通用用Hessian2。

Q7：Dubbo服务超时和重试怎么配置？有哪些坑？

基础配置： 
@DubboReference(timeout = 3000, retries = 2)
private OrderService orderService;
timeout=3000 3秒超时，retries=2 失败后重试2次（共3次调用）。

⚠️ 必知坑：
1. 幂等性：重试只适用于查询类接口！扣库存/扣款接口必须设置 retries=0，否则可能重复扣款。
2. 超时传递：如果A→B→C链路调用，A超时3秒但B调C要5秒——C会被B的超时截断。需要合理设置每段超时。
3. 超时 vs 重试时间：总耗时 = timeout × (retries+1)，三次重试就是9秒。

最佳实践：查询接口 retries=2, timeout=1s；写接口 retries=0, timeout=3s。

Q8：Dubbo服务治理包括哪些能力？

8大服务治理能力：
1. 服务注册与发现：Provider启动后自动注册，Consumer自动发现
2. 负载均衡：5种策略（Random/RoundRobin/LeastActive/ConsistentHash/ShortestResponse）
3. 容错：Failover失败重试 / Failfast快速失败 / Failsafe忽略异常 / Failback定时重发
4. 限流：TpsLimiter控制每秒调用量
5. 降级：服务不可用时返回Mock数据
6. 路由：按标签/机房/IP做流量路由
7. 权重动态调整：不停机上下线机器（调整权重→0→摘除）
8. 监控：调用量/耗时/错误率实时大盘（Dubbo Admin）

一句话："Dubbo不是单纯的RPC框架——它是完整的服务治理平台。"

Q9：Dubbo 3.x相比2.x有哪些核心升级？

三大核心变化：
1. 全新的服务发现模型：从"接口级"升级为"应用级"——注册中心数据量从 O(接口数) 降到 O(应用数)，支持百万级实例。
2. Triple协议：基于HTTP/2 + Protobuf，兼容gRPC，支持流式调用（Streaming）和浏览器直接访问。
3. 云原生增强：原生支持K8s Probes、xDS协议（Istio服务网格），更适合容器化部署。

面试亮点："Dubbo 3用了应用级服务发现，把注册中心压力降了90%——以前每个接口都是独立服务，现在整个应用注册一次。"

Q10：实际项目中 Dubbo 常见问题怎么排查？

Top 5 问题+排查步骤：
1. 调用超时：检查Provider日志→看方法耗时→加大timeout或优化SQL
2. No Provider：检查Provider是否注册到注册中心→Dubbo Admin查看→检查group/version是否匹配
3. 序列化异常：检查DTO类是否实现Serializable→参数中有没有不可序列化对象
4. 线程池满：Provider线程池默认200→检查是否有慢方法阻塞线程→加大threadpool或异步化
5. 调用不均衡：某台机器的负载远高于其他→检查是否LeastActive策略在旧的慢机器上积压→或用Random加权分散
