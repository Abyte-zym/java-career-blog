🛠 Maven + Gradle 构建工具面试10题

2026年Java实习面试中构建工具问题出现率稳定在30%，尤其大厂看重工程化能力。每道题附带面试话术和避坑指南。

1. Maven的生命周期有哪些？常用命令分别对应哪个阶段？

标准答案：Maven有3套生命周期——clean（清理target）、default（核心构建）、site（文档生成）。面试只考default： 
validate → compile → test → package → verify → install → deploy
面试话术："最常用的是clean、compile、test、package、install。mvn clean install = 先清理再打包安装到本地仓库。mvn clean package -DskipTests = 跳过测试打包。"

2. 如何解决Maven依赖冲突？有哪些常见策略？

标准答案：Maven依赖冲突遵循最短路径优先和先声明优先原则。解决策略：
  1. 用 mvn dependency:tree 查看依赖树，定位冲突
  2. 在 <dependencyManagement> 统一声明版本
  3. 用 <exclusions> 排除不需要的传递依赖
  4. 用 <optional>true</optional> 声明可选依赖
面试话术："遇到NoClassDefFoundError或ClassNotFoundException就是依赖冲突。先用dependency:tree定位，再看是排除还是统一版本。最坑的是同一个jar两个版本共存——JVM只会加载其中一个。"

3. Maven和Gradle有什么区别？实际项目怎么选？

标准答案：
维度MavenGradle
配置语法XML（pom.xml）Groovy/DSL（build.gradle）
构建速度较慢（全量编译）快（增量编译+缓存）
灵活性低（约定优于配置）高（脚本化构建）
依赖管理传递依赖+scope传递依赖+bom+版本锁定
流行度企业主流（70%+）新项目增长快（Android标配）
面试话术："简单项目Maven够用，Spring Boot标准也是Maven。Gradle适合Android和多模块大型项目——构建快、脚本灵活。我们课程项目用Maven，个人项目换Gradle尝过鲜。"

4. Maven多模块项目怎么配置？父子模块之间怎么引用？

标准答案：父POM声明 <packaging>pom</packaging>，用 <modules> 列出所有子模块。子模块通过 <parent> 继承父POM，子模块之间通过 <dependency> 互相引用。 面试话术："微服务项目通常按功能分模块——common模块写公共类、api模块放接口定义、service模块写业务。父POM统一管理依赖版本号，子模块只声明不写版本。"

5. Maven的scope有哪些？各自什么场景用？

标准答案： 
compile   — 默认，所有阶段可用（如spring-boot-starter-web）
provided  — 编译+测试需要，运行时由容器提供（如servlet-api）
runtime    — 运行时需要，编译不需要（如mysql-connector-j）
test       — 仅测试阶段可用（如junit、mockito）
system     — 类似provided但jar必须手动指定路径（极少用）
import     — 用在dependencyManagement，导入BOM
面试话术："最常见的坑是把servlet-api写成compile——部署到Tomcat会冲突。Spring Boot内嵌Tomcat不怕，但传统项目Tomcat已提供该jar。"

6. Gradle中的Task是什么？怎么自定义Task？

标准答案：Task是Gradle的最小执行单元。每个构建操作（编译/打包/测试）都是一个Task。Task有依赖关系，Gradle通过DAG（有向无环图）管理执行顺序。 
// 自定义Task示例
task helloWorld {
    dependsOn 'compileJava'
    doLast {
        println '构建完成！'
    }
}
面试话术："Gradle的灵活性体现在Task可以随意自定义。比如在Spring Boot打包后自动推送Docker镜像，写一个Custom DockerTask依赖bootJar就行。"

7. Maven插件机制原理是什么？Spring Boot的Maven插件做了什么？

标准答案：Maven插件绑定到生命周期的某个阶段（phase），到了该阶段自动执行。Spring Boot Maven插件（spring-boot-maven-plugin）做的事：
  1. 打包：把项目打成Fat Jar（包含依赖+内嵌Tomcat）
  2. 启动：mvn spring-boot:run 直接运行
  3. 构建信息：生成build-info.properties
面试话术："Spring Boot打包后的Fat Jar可以直接java -jar运行，不用部署Tomcat——就是spring-boot-maven-plugin的功劳。它把所有依赖和项目代码打包成一个可执行jar。"

8. 什么是BOM（Bill of Materials）？在Maven和Gradle里怎么用？

标准答案：BOM统一管理一组依赖的版本号，避免多个模块各自声明相同版本导致冲突。 
<!-- Maven中导入Spring Cloud BOM -->
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.springframework.cloud</groupId>
      <artifactId>spring-cloud-dependencies</artifactId>
      <version>2024.0.0</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
面试话术："Spring Cloud全系列依赖版本由BOM统一管理——我只需要在dependencyManagement引入spring-cloud-dependencies，后面声明具体依赖时不用写版本号。升级时只改BOM一行。"

9. Maven的仓库体系是什么？中央仓库、私服、本地仓库的关系？

标准答案：查找顺序：本地仓库(~/.m2/repository) → 私服(Nexus) → 中央仓库(Maven Central) → 远程仓库。企业内网通常搭建Nexus私服，统一管理内部包+代理外部包。 面试话术："私服的好处：①内网速度快 ②内部发布私有jar ③统一管控第三方依赖安全。配置文件在settings.xml里配mirror指向私服。"

10. Gradle的增量构建是什么？怎么利用它加速CI/CD？

标准答案：Gradle的增量构建：只编译修改过的文件+依赖这些文件的类。核心机制：
  1. 输入/输出快照：记录每个Task的上次输入输出
  2. 缓存对比：如果输入没变，直接跳过该Task
  3. 跨构建缓存：多个项目共享编译缓存（Build Cache）
面试话术："Gradle增量构建比Maven快的原因：它追踪文件变更→只编译受影响的部分→跳过没变的。CI/CD里build cache持久化到远程缓存服务，二次构建秒级完成。"
