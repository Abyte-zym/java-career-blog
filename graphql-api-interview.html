Q1：GraphQL和REST有什么区别？什么场景用哪个？

核心区别：
| REST | GraphQL |
| 多端点（/users, /posts, /comments） | 单端点（POST /graphql） |
| 服务端定义返回字段 | 客户端指定返回字段 |
| 容易Over-fetching/Under-fetching | 精确返回，不多不少 |
| HTTP缓存天然支持 | 需自己处理缓存 |
| 学习成本低 | Schema类型系统学习曲线 |

选型：简单CRUD→REST。复杂关联查询、移动端多端不同数据需求→GraphQL。两者可共存。

Q2：GraphQL如何解决N+1查询问题？

问题场景：查询10个作者，每个作者有N本书——1次查作者 + 10次查书 = 11次SQL。

解决方案：DataLoader批处理 
private DataLoader> bookLoader = 
    DataLoader.newDataLoader(ids -> 
        CompletableFuture.supplyAsync(() -> 
            bookRepo.findByAuthorIds(ids)  // 一次SQL: WHERE author_id IN (1,2,3,...)
        )
    );
原理：DataLoader收集同一帧内的所有key→合并成一条SQL→结果按key分发回调用者。MySQL的 WHERE author_id IN (1,2,3,...) 一条查询替代N条。

Q3：GraphQL Schema怎么定义？核心类型有哪些？

Schema示例： 
type Query {
    user(id: ID!): User
    users(page: Int, size: Int): [User!]!
}

type Mutation {
    createUser(input: CreateUserInput!): User!
    deleteUser(id: ID!): Boolean!
}

type Subscription {
    userCreated: User!
}

type User {
    id: ID!
    name: String!
    email: String
    posts: [Post!]!  # 关联查询
    createdAt: String!
}
三大根类型：Query(读)、Mutation(写/删/改)、Subscription(实时推送)。! 表示非空。

Q4：GraphQL Subscription是什么？和WebSocket什么关系？

定义：客户端订阅某个事件，服务端有变化时主动推送。底层走WebSocket。

工作流程：
1. 客户端发送订阅请求：subscription { userCreated { id name } }
2. 服务端建立WebSocket长连接
3. 每次有用户创建→服务端推送给所有订阅者

适用场景：实时通知、聊天消息、股票行情、订单状态变更。

Q5：Spring Boot怎么集成GraphQL？

依赖（Spring for GraphQL）： 
implementation 'org.springframework.boot:spring-boot-starter-graphql'
定义Controller： 
@Controller
public class UserGraphQLController {
    
    @QueryMapping
    public User user(@Argument Long id) {
        return userService.findById(id);
    }
    
    @MutationMapping
    public User createUser(@Argument CreateUserInput input) {
        return userService.create(input);
    }
    
    @SubscriptionMapping
    public Flux userCreated() {
        return userService.userCreatedStream();
    }
    
    @SchemaMapping(typeName = "User", field = "posts")
    public List posts(User user, @Argument int limit) {
        return postService.findByAuthorId(user.getId(), limit);
    }
}
Schema文件：src/main/resources/graphql/schema.graphqls 与Java代码自动映射。

Q6：GraphQL的安全性怎么保证？有哪些攻击方式？

5种防御：
1. 深度限制：防止循环嵌套查询炸服——graphql.servlet.max-query-depth=5
2. 复杂度限制：给每个字段分配成本，查询总成本超阈值拒绝
3. 速率限制：单端点容易被滥用，需要per-query或per-IP限流
4. 查询白名单：生产环境只允许预先注册的persisted queries
5. 超时控制：防止慢查询资源耗尽

面试提到graphql-java-kickstart + voyager 做SDL校验是加分项。

Q7：GraphQL的缓存怎么做？和REST有什么不同？

REST的缓存天然简单：每个URL一个资源，CDN+浏览器直接缓存。

GraphQL的缓存挑战：单端点POST，CDN无法按query内容缓存。

解决方案：
1. 持久化查询（Persisted Queries）：查询ID→GET /graphql?id=xxx → 可被CDN缓存
2. DataLoader层缓存：同一请求内去重
3. 应用层缓存：Redis缓存DataLoader的批量查询结果
4. 客户端缓存：Apollo Client/Relay自动归一化缓存（Normalization Caching）

Q8：GraphQL的错误处理怎么做？和REST的HTTP状态码有什么不同？

核心差异：GraphQL总返回200 OK。错误放在响应体的 errors 数组里。

响应示例： 
{
  "data": { "user": null },
  "errors": [{
    "message": "User not found",
    "extensions": {
      "code": "NOT_FOUND",
      "classification": "DataFetchingException"
    }
  }]
}
面试说："GraphQL允许部分成功——10个字段里8个成功2个失败，不会全滚。这对移动端弱网环境很友好。"

Q9：GraphQL怎么处理文件上传？

GraphQL原生不支持文件上传——因为它是JSON协议。需要扩展：

方案1（graphql-multipart）：
mutation ($file: Upload!) {
    uploadAvatar(file: $file) { url }
}
用 multipart/form-data 编码，文件二进制+GraphQL查询混传。

方案2（实践中更推荐）：
单独REST端点 POST /upload 上传文件→返回URL→再调GraphQL mutation把URL关联到用户。这样CDN缓存、断点续传、进度条都容易做。

Q10：你在项目中用过GraphQL吗？没用过怎么回答？

诚实但展示学习能力：
"目前项目用的是REST，但我主动了解过GraphQL。知道它解决的核心问题是客户端数据需求的灵活性——移动端不同页面需要不同字段组合，REST要调两三个接口，GraphQL一个查询搞定。

我对DataLoader的批处理机制印象深刻——它用Event Loop收集同帧key然后合并SQL，和我们秒杀项目里消息队列批量消费的思路很像。

如果团队有GraphQL需求，我有信心快速上手——Schema类型系统和Spring的注解驱动开发思路一致。"

关键：不提"没做过"，只提"我了解过+愿意学+有相似经验"。
