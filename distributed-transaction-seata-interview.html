Q1：分布式事务为什么需要？本地事务ACID在微服务架构下有什么问题？

一句话：一个业务操作跨多个微服务，每个服务有自己的数据库。本地事务只保证自己的ACID，无法保证全局一致性。

例子：下单操作→扣库存(库存服务)+扣余额(账户服务)+生成订单(订单服务)。如果扣库存成功但扣余额失败——扣了库存没收到钱，系统数据不一致。

核心矛盾：CAP定理——无法同时满足一致性(C)、可用性(A)、分区容错性(P)。分布式系统必选P，在C和A之间做权衡。多数业务场景选择AP(最终一致性)。

Q2：2PC(两阶段提交)原理是什么？有什么致命缺陷？

两阶段：
阶段一(Prepare)：协调者向所有参与者发"准备提交"请求→各参与者执行事务但不提交→返回YES/NO
阶段二(Commit/Rollback)：全部YES→发"提交"→全部NO则发"回滚"

致命缺陷：
1. 同步阻塞：Prepare阶段锁资源，第二阶段结束前不释放→高并发下锁等待
2. 单点故障：协调者挂了→所有参与者阻塞等待
3. 数据不一致：第二阶段如果协调者只发了部分commit→部分参与者提交、部分没收到→数据不一致

Q3：3PC怎么改进2PC？

3PC = CanCommit + PreCommit + DoCommit

1. CanCommit：协调者问"能提交吗？"——只检查不执行，无锁
2. PreCommit：执行事务+写undo/redo日志
3. DoCommit：最终提交

改进点：引入超时机制——PreCommit阶段超时→自动提交(因为已经确认了CanCommit)。避免了2PC协调者挂掉导致的无限阻塞。

代价：多一轮通信，性能更低。实际生产中很少用3PC。

Q4：TCC模式原理？Try/Confirm/Cancel各做什么？

TCC = Try + Confirm + Cancel
Try: 预留资源（冻结库存、冻结余额）→ 不真正扣
Confirm: 真正执行（扣减冻结的库存、转账）→ 幂等
Cancel: 回滚（解冻库存、退款）→ 补偿操作
例子（下单）：
- Try：库存服务冻结1件商品、账户服务冻结100元
- Confirm：库存服务扣减冻结的1件、账户服务扣100元
- Cancel：库存服务解冻1件、账户服务退100元

优缺点：每个服务需要提供Try/Confirm/Cancel三个接口→业务侵入性强。但性能好（无锁）、一致性高。

Q5：Seata AT模式怎么工作的？和XA有什么区别？

Seata AT = 自动补偿型分布式事务
1. 一阶段(执行业务SQL)：Seata拦截SQL→记录undo_log→业务SQL正常提交
2. 二阶段(提交/回滚)：全局提交→异步删undo_log；回滚→读undo_log反向补偿

Seata AT vs XA：
| | XA | Seata AT |
| 锁释放 | 二阶段才释放 | 一阶段就释放 |
| 性能 | 差 | 较好 |
| 侵入性 | 无 | 无(自动代理) |
| 一致性 | 强一致 | 最终一致 |

适用场景：传统单体拆分微服务的第一阶段——改了代码结构但不想改业务逻辑时用Seata AT。

Q6：可靠消息最终一致性怎么实现？RocketMQ事务消息原理？

流程：
1. 生产者发半消息(half message)——消费者看不到
2. 执行本地事务（如扣库存）
3. 本地事务成功→发commit→消费者可见；失败→发rollback→消息删除
4. 如果生产者挂了没回传→MQ定期回查生产者本地事务状态

你的秒杀项目里：Redis扣库存成功后→发RocketMQ消息→订单服务消费消息落库。如果订单落库失败→利用MQ重试+死信队列兜底。

面试加分：提到消息表+定时对账双保险——MQ挂了也能最终一致。

Q7：最大努力通知方案是什么？和可靠消息有什么区别？

最大努力通知：发起方执行完本地事务→异步通知下游→通知失败就反复重试(指数退避)→超过最大重试次数还失败→记录到失败表→人工介入。

vs 可靠消息：
| | 可靠消息 | 最大努力通知 |
| 一致性保证 | 消息不丢(MQ持久化) | 尽力而为 |
| 适用场景 | 核心交易链路 | 非关键通知(短信/邮件) |
| 成本 | 需要MQ | 简单HTTP+重试即可 |

一句话：订单状态变更通知用户——用最大努力通知。扣款——用可靠消息。

Q8：SAGA模式是什么？什么场景用SAGA而不是TCC？

SAGA = 长事务拆成一串本地事务，每个有对应的补偿操作
正向: 创建订单 → 扣库存 → 扣余额 → 发货
补偿: 取消订单 ← 补库存 ← 退款   ← (不可逆)
SAGA vs TCC：
- TCC：资源预留→执行。适合资金/库存等可冻结的场景
- SAGA：直接执行→如果后面失败逐级补偿。适合长流程、不可逆操作（如已发货无法取消）

两种实现：
- 编排式(Choreography)：每个参与者收到事件自己决定干什么——松耦合但流程不可见
- 协调式(Orchestration)：一个协调者统一指挥——流程清晰但协调者成瓶颈

Q9：Seata的AT模式和TCC模式各适合什么场景？

选型决策表：
| 场景 | 推荐 | 原因 |
| 单体拆微服务初期 | AT | 零侵入，改配置就搞定 |
| 高并发资金类 | TCC | 无锁，性能好 |
| 简单CRUD + 一个全局事务 | AT | 够用 |
| 复杂业务逻辑 + 跨多个服务 | TCC | 补偿逻辑自定义 |
| 已有代码不想改 | AT | @GlobalTransactional一个注解 |

你的项目：秒杀系统用的是可靠消息最终一致性(RocketMQ)——比Seata更轻量。因为秒杀追求的是吞吐量，不是强一致。

Q10：面试官问"分布式事务你们项目怎么做的"——满分回答模板

模板（3层递进）：
1. 先定场景："我们秒杀系统下单需要跨库存服务和订单服务，存在分布式事务问题"
2. 说方案+原因："我们用RocketMQ事务消息实现最终一致性——Redis扣库存成功后发半消息，订单服务消费落库。选这个而不是Seata是因为秒杀追求吞吐量，强一致会拖慢整个链路"
3. 兜底方案："MQ重试+死信队列处理临时故障，定时对账任务跑历史数据修复极端异常"

这样答面试官就知道你：
✅ 理解分布式事务的几种方案
✅ 能根据场景做技术选型
✅ 考虑了兜底方案
✅ 不是说"我们用了XX"而是"我们为什么选XX"
