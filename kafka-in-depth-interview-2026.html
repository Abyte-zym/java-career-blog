Q1：Kafka为什么这么快？零拷贝是什么？

四大原因：
1. 顺序写入磁盘：Kafka把消息顺序追加到日志文件末尾，磁盘顺序写速度(600MB/s+)远高于随机写(100KB/s)。
2. 零拷贝(Zero-Copy)：消费时直接通过sendfile()系统调用，数据从磁盘→内核PageCache→网卡，完全不过用户态、不复制。
3. PageCache缓存：利用OS的页面缓存，热数据直接读内存。
4. 批量发送+压缩：Producer将多条消息打包成batch，用snappy/gzip压缩后一次发送。

面试多说一句：零拷贝比传统read→write少2次CPU拷贝+2次上下文切换，吞吐量提升3-5倍。

Q2：Kafka的分区策略有哪些？

三种分区策略：
1. 指定分区：Producer消息带partition号→直接发到指定分区。
2. Key哈希：没有partition但有key→hash(key) % partition_count。相同key永远进同一分区=消息有序。
3. 轮询(Round-Robin)：既没有partition也没有key→均匀分布到所有分区。

// 自定义分区器：按订单金额分发
public class AmountPartitioner implements Partitioner {
    public int partition(String topic, Object key, byte[] kb, Object val, byte[] vb, Cluster c) {
        long amount = ((Order) val).getAmount();
        return (int) (amount % c.partitionCountForTopic(topic));
    }
}

Q3：ISR是什么？为什么重要？

ISR = In-Sync Replicas（同步副本集）：当前与Leader保持同步的Follower集合。

核心作用：Kafka不要求所有副本都同步才commit——只等ISR里的副本确认。这样在性能和数据可靠性间取得平衡。

Follower被踢出ISR的条件：超过replica.lag.time.max.ms(默认30秒)没同步。

面试加分：配置min.insync.replicas=2可以保证至少2个副本确认才返回ack——否则会出现Leader挂了数据丢失的场景。

Q4：Kafka如何实现Exactly-Once语义？

三层保障：
1. 幂等生产者（Idempotent Producer）：给每个Producer分配PID，每条消息带Sequence Number。Broker发现重复SN直接丢弃→单分区内不重复。
2. 事务（Transactions）：跨分区原子写入。Producer调用initTransactions()→beginTransaction()→发送→commitTransaction()。
3. Consumer配合：设置isolation.level=read_committed，只读已提交的事务消息，屏蔽未提交的脏数据。

配置： 
// 幂等生产者
enable.idempotence=true
// 事务
transactional.id=tx-001

Q5：消费者组重平衡(Rebalance)是怎么回事？怎么避免？

触发条件：消费者增减/分区变化/消费者超时没有心跳(默认session.timeout.ms=45s)。

三种策略： Range（默认）：按分区序号均分，可能不均。 RoundRobin：依次轮流分配，均匀。 Sticky：尽量保持之前的分配，减少不必要的迁移。

避免频繁Rebalance： 1. 调大session.timeout.ms（如120s） 2. 调大max.poll.interval.ms（两次poll间隔，防止处理太慢被踢） 3. 使用静态成员(Static Membership)：group.instance.id固定组成员ID，重启后不触发Rebalance

Q6：Kafka如何保证消息不丢失？

生产者端： - acks=all（或-1）：所有ISR确认才返回 - retries=Integer.MAX_VALUE：无限重试 - enable.idempotence=true：防重
Broker端： - min.insync.replicas>=2 - unclean.leader.election.enable=false：禁止落后副本竞选Leader
消费者端： - enable.auto.commit=false：手动提交offset - 先处理消息再commit（不是先commit再处理）

一句话：acks=all + 手动提交 + ISR>=2 = 三重保险。

Q7：Kafka中partition和consumer的对应关系？

核心规则：同一消费者组内，一个分区只能被一个消费者消费，但一个消费者可以消费多个分区。

推论： - 消费者数 > 分区数 → 多余的消费者闲着 - 分区数 > 消费者数 → 每个消费者扛多个分区 - 最佳配置：分区数 = 消费者数 x 2（留扩展空间）

面试场景题："QPS要10万，一条消息处理10ms，怎么设计？"
→ 10万QPS × 10ms = 1000分区，15个消费者实例，每台扛67个分区。

Q8：Kafka和RocketMQ选哪个？

快速对比：
| 维度 | Kafka | RocketMQ | | 事务消息 | 较新 | ✅ 原生支持，金融级 | | 延迟消息 | 无内置 | ✅ 18个级别 | | 吞吐量 | ✅ 更高 | 中等 | | 消息回溯 | 按offset | 按时间 | | 运维 | 复杂 | ✅ 阿里出品，好用 |
选型口诀：高吞吐大数据→Kafka；金融/订单无丢失→RocketMQ；中小团队→RocketMQ省心。
你的秒杀项目：选RocketMQ——因为需要事务消息确保下单一致性。

Q9：Kafka的消息怎么保证顺序？

Kafka只保证单个分区内有序。

全局有序： - 只用1个分区（牺牲并行度） - 或用同一个key让同业务进同一分区

生产者端： - max.in.flight.requests.per.connection=1（幂等开启后自动设5也行）
消费者端： - 单线程消费一个分区 - 多线程消费→维护内存队列保证有序

面试亮点：提到"Kafka用key哈希+单分区消费保证业务级有序，RocketMQ用消息组(MessageGroup)做全局有序更简单。"

Q10：Kafka Controller是什么？怎么选举的？

Controller是集群中某个特殊的Broker，负责管理分区和副本的状态。

核心职责： - 分区Leader选举 - 监听Broker上下线 - ISR变更通知 - 分区重分配
选举机制： 所有Broker启动时在ZooKeeper的/controller节点上抢注册——谁先创建临时节点谁就是Controller。Controller挂了→临时节点自动删除→剩下的Broker重新抢。

面试多说：Kafka 3.x已移除ZooKeeper依赖(改用KRaft)，Controller选举基于Raft协议，更简单。
