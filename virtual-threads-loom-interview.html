Q1：什么是虚拟线程？Java 21引入它解决了什么问题？

虚拟线程（Virtual Thread）：JDK 21引入的轻量级线程，由JVM而非操作系统管理。一个操作系统线程可以承载数千个虚拟线程。

解决的问题：传统平台线程（Platform Thread）是1:1映射到OS线程的，创建成本高（约1MB栈空间），线程池再大也有限。虚拟线程解决了"每个请求一个线程"模式下OS线程不够用的问题。

一句话总结：虚拟线程让Java可以用"每个任务一个线程"的方式写代码，但实际只消耗很少的OS资源。

代码示例： 
// 创建10万个虚拟线程——传统线程根本不可能
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            Thread.sleep(100); // 模拟IO
            return "done";
        });
    }
}

Q2：平台线程和虚拟线程的本质区别是什么？

对比表：
| 维度 | 平台线程 | 虚拟线程 | |------|---------|---------| | 调度者 | OS内核 | JVM（ForkJoinPool） | | 栈大小 | ~1MB | 按需增长，初始几百字节 | | 创建成本 | 高（系统调用） | 极低（堆内存对象） | | 上下文切换 | 内核态切换 | 用户态切换 | | 数量上限 | ~几千 | 数百万 | | 适用场景 | 计算密集型 | IO密集型 |
核心区别：平台线程在阻塞（sleep/IO/锁等待）时OS线程也被阻塞。虚拟线程阻塞时Carrier Thread被释放去执行其他虚拟线程——IO密集型吞吐量飙升。

Q3：虚拟线程的调度原理是什么？什么是Carrier Thread？

Carrier Thread（载体线程）：真正执行虚拟线程的平台线程。JVM用ForkJoinPool管理一组Carrier Thread（默认数量=CPU核心数）。

调度流程：
1. 虚拟线程被mount（挂载）到某个Carrier Thread上执行
2. 虚拟线程遇到阻塞操作（IO/sleep/lock）→ 自动unmount（卸载）
3. Carrier Thread立刻去执行另一个虚拟线程
4. 阻塞结束后→虚拟线程重新mount到任意可用的Carrier Thread继续执行

面试必须提的关键词："mount/unmount"、"continuation（续体）"、"用户态切换"。

Q4：虚拟线程在什么情况下会被pin（钉住）？

Pin指虚拟线程无法从Carrier Thread上unmount，导致Carrier Thread被长期占用。

Pin的常见场景（面试高分点！）：
1. synchronized块/方法内阻塞——这是最常见的pin场景
2. 调用native方法或JNI时
3. 获取某些特定对象（如Class对象）的monitor时

解决方案：把synchronized改成 ReentrantLock。JDK团队未来版本会消除synchronized的pin限制。

一句亮点："如果大量虚拟线程在synchronized里阻塞IO，线程池会被耗尽。生产环境默认用ReentrantLock替代synchronized。"

Q5：Spring Boot 3.2+如何启用虚拟线程？

# application.yml
spring:
  threads:
    virtual:
      enabled: true  # 一行配置，Tomcat/Jetty/Undertow的请求线程全变虚拟线程
也可以手动配置： 
@Configuration
public class VirtualThreadConfig {
    @Bean(TaskExecutionAutoConfiguration.APPLICATION_TASK_EXECUTOR_BEAN_NAME)
    public AsyncTaskExecutor asyncTaskExecutor() {
        return new TaskExecutorAdapter(Executors.newVirtualThreadPerTaskExecutor());
    }
    
    @Bean
    public TomcatProtocolHandlerCustomizer protocolHandlerVirtualThreadExecutorCustomizer() {
        return handler -> handler.setExecutor(Executors.newVirtualThreadPerTaskExecutor());
    }
}
效果：一行配置，Controller处理请求、@Async异步任务、定时调度全走虚拟线程。

Q6：虚拟线程和线程池的区别？什么时候用线程池vs虚拟线程？

线程池的问题：池大小固定。如果所有线程都在等IO→再来的请求只能排队→CPU空闲但系统卡。

虚拟线程的优势：不需要池。来一个请求创一个虚拟线程，IO阻塞时自动切换。池大小无限（取决于内存）。

选择原则：
- IO密集型（网络请求、数据库查询、RPC调用）→ 虚拟线程——阻塞时释放Carrier Thread
- CPU密集型（加密、压缩、图像处理）→ 平台线程池——虚拟线程不需要切换，但没必要
- 需要限制并发数 → 信号量（Semaphore）包裹虚拟线程——别用线程池限制虚拟线程数

Q7：虚拟线程的ThreadLocal怎么处理？

ThreadLocal仍是线程级别的存储：虚拟线程支持ThreadLocal，但值只在同一个虚拟线程内可见。

潜在问题：虚拟线程数量巨大（百万级），每个都有ThreadLocal会占用大量内存。

解决方案：
1. 改用 ScopedValue（JDK 21 预览，JDK 22 正式）——不可变、生命周期明确的线程本地替代
2. 确保ThreadLocal用完后clear()，避免内存泄漏

面试一句答对："虚拟线程支持ThreadLocal，但推荐迁移到ScopedValue以避免百万虚拟线程的内存膨胀。"

Q8：Tomcat使用虚拟线程后，max-connections和max-threads怎么设置？

虚拟线程时代，max-threads不再限制并发请求数。

推荐配置： 
server:
  tomcat:
    max-connections: 10000   # 最大连接数（仍有效）
    threads:
      max: 200   # 仅影响连接器的接收线程，不影响请求处理
spring:
  threads:
    virtual:
      enabled: true  # 请求处理走虚拟线程，不受max-threads限制
关键认知：启用虚拟线程后，理论上可以同时处理几万个请求（受内存限制），不需要设置max-threads上限。但要用Semaphore限制对数据库等后端的并发数。

Q9：虚拟线程对数据库连接池有什么影响？

这是一个面试陷阱题。

问题：如果10000个虚拟线程同时访问数据库，而连接池只有20个连接→9990个虚拟线程阻塞等待连接→但虚拟线程阻塞会释放Carrier Thread→系统不崩，只是数据库成了瓶颈。

正确做法：用 Semaphore 限制并发数据库请求数： 
Semaphore dbSemaphore = new Semaphore(20);
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        dbSemaphore.acquire();
        try {
            jdbcTemplate.query(...); // 只有20个并发能进这里
        } finally {
            dbSemaphore.release();
        }
    });
}
面试金句："虚拟线程让IO密集不再卡CPU，但会暴露数据库/下游服务的瓶颈。需要给每个瓶颈单独加并发控制。"

Q10：在高并发秒杀项目中，虚拟线程能替代RocketMQ削峰吗？

不能。虚拟线程和MQ是不同维度的事。

虚拟线程解决的是：IO阻塞时的CPU利用率问题——让CPU不在等网络/磁盘时闲着。
MQ解决的是：流量削峰、异步解耦、最终一致性——让数据库不在流量洪峰下崩溃。

实战公式：
秒杀请求 → 虚拟线程处理（高并发接收）
         → Semaphore限流（控制进MQ的速度）
         → RocketMQ削峰（保护数据库）
         → 虚拟线程消费（高并发处理订单）
面试总结："虚拟线程让每个请求处理更快、资源利用率更高，但不能替代MQ削峰。两者配合才是最优解。"
