Q1：什么是虚拟线程？和平台线程有什么区别？

虚拟线程（Virtual Thread）：Java 21正式GA的轻量级线程，由JVM管理而非OS。几百万个虚拟线程只占几MB内存。

核心区别表：
| 平台线程 | 虚拟线程 | | OS线程1:1 | JVM管理M:N映射 | | 创建慢(～1ms) | 创建极快(～1μs) | | 一个线程～1MB栈 | 一个虚拟线程～0.2KB栈(按需分配) | | 最多几千个 | 可上百万 | | 线程池复用 | 用完就扔，不池化 |

一句话：虚拟线程=廉价线程。不再需要线程池那一套，直接 newVirtualThreadPerTaskExecutor 就行。

Q2：虚拟线程怎么创建？

三种方式：
// 1. Thread类直接创建
Thread vt = Thread.startVirtualThread(() -> {
    System.out.println("Hello from virtual thread");
});

// 2. Thread.Builder（推荐）
Thread vt2 = Thread.ofVirtual()
    .name("my-vt-1")
    .start(() -> doWork());

// 3. 虚拟线程执行器（最常用）
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> doWork());
    executor.submit(() -> doAnotherWork());
}
最佳实践：newVirtualThreadPerTaskExecutor() ——每个任务一个虚拟线程，无需调参，无需池化。

Q3：什么时候该用虚拟线程？什么时候不该用？

✅ 该用的场景（IO密集）：
- HTTP请求等待响应
- 数据库查询
- 读写文件
- 调用其他微服务

❌ 不该用的场景（CPU密集）：
- 纯计算任务（如图像处理、加密）
- 需要长时间持有synchronized锁的任务（会pin住载体线程）

面试时这样说："虚拟线程解决的是吞吐量问题而非延迟问题。一个请求等IO时让出CPU给其他虚拟线程——所以IO密集场景受益最大。"

Q4：Spring Boot 3.x怎么用虚拟线程？

一行配置开启：
spring.threads.virtual.enabled=true  # Spring Boot 3.2+
加了这个，Tomcat的请求处理线程、@Async方法、RabbitMQ listener 全自动切到虚拟线程。

面试加分：提一句RestClient和WebClient配合虚拟线程——"原来用WebClient异步回调很复杂，现在用RestClient同步写法+虚拟线程，代码简洁且吞吐量一样高"。

Q5：什么是Pinned Thread（钉住线程）？怎么排查？

Pinned Thread：当虚拟线程进入 synchronized 块 或调用 native 方法 / Foreign Function 时，它会把底层的载体线程"钉住"——这个虚拟线程被阻塞时，载体线程也被阻塞，无法切换到其他虚拟线程。

排查命令：
# JFR监控Pinned Thread事件
java -XX:StartFlightRecording:filename=recording.jfr \
     -XX:+UnlockDiagnosticVMOptions \
     -XX:+EnablePinnedThreadEvent

# 或用 jcmd 实时查看
jcmd <pid> Thread.dump_to_file -format=json threaddump.json
解决方案：把 synchronized 替换成 ReentrantLock 即可解除Pinned。Java 24+已优化，native调用不再全部pin。

Q6：虚拟线程和协程（Kotlin Coroutine / Go Goroutine）有什么区别？

对比表：
| 特性 | Java虚拟线程 | Go Goroutine | Kotlin协程 | |------|------------|-------------|-----------| | 调度者 | JVM ForkJoinPool | Go Runtime | Kotlin编译器+库 | | 染色 | 无染色函数 | 无染色函数 | ⚠️ 函数染色(suspend) | | API改动 | 零改动 | 零改动 | 需重构为suspend | | 抢占 | 协作式(IO自动让出) | 协作式+抢占式 | 协作式 |

一句话：虚拟线程最大优势是"零代码改动"——现有的同步代码原封不动，换个Executor就从线程池升级到虚拟线程。

Q7：StructuredTaskScope是什么？怎么用？

核心概念：结构化的并发——把多个子任务的生命周期绑定到一个作用域（Scope），保证所有子任务在Scope关闭前结束。

try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    // 同时发起3个调用
    Future<User> user = scope.fork(() -> fetchUser());
    Future<Order> order = scope.fork(() -> fetchOrder());
    Future<Review> review = scope.fork(() -> fetchReview());

    scope.join();           // 等全部完成或任一失败
    scope.throwIfFailed();  // 有失败的抛异常

    return new Result(user.resultNow(), order.resultNow(), review.resultNow());
} // 自动关闭scope，清理线程
面试时这样说："原来用CompletableFuture.allOf()组合多个异步调用，异常处理和超时控制很手动。StructuredTaskScope把3个IO调用结构化——一个join等全部，任一个失败自动取消其他两个，作用域结束自动清理线程资源。"

Q8：虚拟线程的ThreadLocal行为有什么变化？

虚拟线程也支持ThreadLocal，但不建议大量使用：
- 虚拟线程生命周期短，频繁创建/销毁下 ThreadLocal 的 set/remove 不划算
- 推荐用 ScopedValue（Java 21 Preview，22正式GA）替代

// 老写法（避免）
static final ThreadLocal<User> currentUser = new ThreadLocal<>();

// 新写法（推荐）
static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

ScopedValue.where(CURRENT_USER, user)
    .run(() -> doWork());
ScopedValue优势：不可变、自动传播到子虚拟线程、不需要clear（作用域结束自动清理）。

Q9：虚拟线程在实际项目中遇到什么坑？

3个真实坑：
1. 连接池耗尽：虚拟线程让并发量暴增→数据库连接池不够用。解决：调大连接池+用连接池限流（Semaphore）
2. synchronized Pinned阻塞：老代码的synchronized块导致载体线程被钉住→改为ReentrantLock
3. ThreadLocal内存膨胀：百万级虚拟线程各自有ThreadLocal→内存爆。解决：切ScopedValue

面试这样说："我团队迁移到虚拟线程后，吞吐量提升了8倍，但也踩过连接池耗尽的坑——虚拟线程让并发从200飙升到2000，传统连接池瞬间打满。"

Q10：你简历上秒杀项目用了虚拟线程吗？如果用了会怎么改？

回答模板（即使没真用也能说）：

"秒杀项目原来用标准线程池处理下单请求，如果要迁移到虚拟线程，我会做三件事：
1. 把Tomcat请求处理切到虚拟线程——spring.threads.virtual.enabled=true 一行搞定
2. RocketMQ消费者的Listener改为虚拟线程执行——减少消费延迟
3. Redis调用的阻塞IO被虚拟线程自动让出——整体吞吐量预期提升3-5倍

但需要注意：同步调用Redis Lua脚本那部分用了synchronized保证库存一致——换成ReentrantLock就不会pinned了。"

面试官心思：这道题考察"学以致用"——不只看你背概念，还看你能不能把新技术落到自己的项目上。
