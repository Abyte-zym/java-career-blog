Q1：为什么要用JMH？不用System.currentTimeMillis()测性能？

JMH（Java Microbenchmark Harness）是OpenJDK官方基准测试工具。

直接用System.currentTimeMillis()的致命缺陷：
1. JVM预热：JIT编译器在方法执行一定次数后才触发编译。第一次跑100ms，跑1000次后可能只有10ms。手写测试拿到的可能是预热前的慢数据。
2. 死代码消除（Dead Code Elimination）：JIT检测到计算结果没被使用，直接删掉整段代码。你测的不是"执行时间"而是"0时间"。
3. 循环展开/向量化：JIT可能把你手写的循环优化掉。
4. GC干扰：测试途中Full GC暂停50ms，你的System.currentTimeMillis()全算进去了。

JMH做了什么：自动预热(fork=3, warmup=5)、Blackhole防止死代码消除、隔离GC影响、多线程安全测量。

Q2：JMH有哪些核心注解？各干什么？

6个核心注解：
@BenchmarkMode(Mode.AverageTime)  // 测量平均时间
@OutputTimeUnit(TimeUnit.NANOSECONDS)
@Warmup(iterations = 5, time = 1)  // 预热5轮，每轮1秒
@Measurement(iterations = 10, time = 1)  // 正式测量10轮
@Fork(3)  // Fork 3个独立JVM进程（避免JVM优化跨测试传播）
@State(Scope.Benchmark)  // 共享状态
Mode枚举：Throughput（吞吐量）、AverageTime（平均时间）、SampleTime（采样分布）、SingleShotTime（冷启动）、All（全部测）。

Q3：StringBuilder真的比String快吗？用JMH怎么验证？

测出来才敢说： 
@Benchmark
public String stringConcat() {
    String s = "";
    for (int i = 0; i < 100; i++) s += i;
    return s;  // 每次+=都new String，100次=100个对象
}

@Benchmark
public String stringBuilder() {
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < 100; i++) sb.append(i);
    return sb.toString();  // 只产生1个对象
}
实测结果（JMH）：StringBuilder比String.concat快约200-400倍。String.concat每次循环创建新String→100次=100个对象→GC压力大。StringBuilder内部是可扩容char数组，不产生中间对象。

但：Javac会对简单String拼接做编译优化，单行String a="a"+"b"+"c"会被优化成StringBuilder。复杂场景（循环内拼接）才需要手写StringBuilder。

Q4：怎么排查线上CPU突然飙到100%？

五步排查法：
1. top -Hp PID — 找到CPU最高的线程TID
2. printf '%x\n' TID — TID转十六进制
3. jstack PID | grep 十六进制 -A 30 — 定位到具体代码行
4. Arthas火焰图：profiler start --duration 60 生成火焰图，一眼看出CPU热点
5. 常见原因：死循环/正则回溯/频繁Full GC/JSON序列化大对象/数据库慢查询阻塞线程池

面试时提Arthas直接加分——比只说jstack/jmap高级。

Q5：接口响应从200ms突然变成2s，怎么定位？

分层排查，从外到内：
1. 网络层：ping/curl -w "@curl-format.txt" 排除网络延迟
2. Nginx层：看access.log的request_time vs upstream_time，判断是Nginx慢还是后端慢
3. 应用层：Arthas trace trace com.xx.Controller methodName 打印每个方法耗时，秒级定位
4. 数据库层：慢查询日志 → EXPLAIN → 缺失索引？锁等待？
5. 缓存层：Redis慢了？redis-cli --latency 看延迟。key过期导致缓存击穿？
6. JVM层：jstat -gcutil PID 1000 看GC频率

Arthas trace是最强武器——一条命令看出整条调用链上每个方法的耗时。

Q6：full gc过于频繁怎么处理？

三步走：
1. 先看现象：jstat -gcutil PID 1000 — FGC/YGC次数、各代内存使用率
2. 找原因：jmap -histo:live PID | head -20 — 哪个类实例最多？可能有内存泄漏
3. 解决方案：
- 代码泄漏：把static Map改成WeakHashMap
- 堆太小：-Xmx调大
- 大量短命对象：调大Young区
- MetaSpace满：-XX:MaxMetaspaceSize调大
- ThreadLocal没remove：改代码

面试必杀技：说"我会先看jstat确认是Young GC频繁还是Full GC频繁，两者原因不同。Young GC频繁→调大Eden区；Full GC频繁→dump堆分析"。

Q7：Arthas用过吗？怎么用的？

生产环境四大场景：
1. dashboard — 实时看线程、内存、GC，类似top但针对JVM
2. trace com.xx.Service method — 打印方法调用链每个节点耗时，比日志高效10倍
3. watch com.xx.Service method '{params, returnObj, throwExp}' — 看方法入参出参，不重启不改代码
4. profiler start --duration 60 — 生成CPU火焰图，一眼看出哪个方法吃CPU

面试官追问"用过吗"：答"用过，Arthas是阿里开源的Java诊断工具，trace命令在生产上帮我快速定位过接口慢的问题，不用加日志不用重启——监控工具和诊断工具的区别"。

Q8：怎么用JMH验证@synchronized和ReentrantLock的性能差异？

设计对比基准测试： 
@Benchmark
@Group("sync")
public void synchronizedMethod() {
    synchronized (lock) { counter++; }
}

@Benchmark
@Group("lock")
public void lockMethod() {
    lock.lock();
    try { counter++; } finally { lock.unlock(); }
}
结论（JDK 6+）：synchronized经过锁升级（偏向→轻量→重量）后，和ReentrantLock性能基本持平。但ReentrantLock提供：tryLock(超时)、公平锁、Condition多条件等待、可中断——这些synchronized做不到。

选型：简单场景用synchronized（代码简洁），需要高级特性用ReentrantLock。

Q9：Spring Boot启动慢怎么排查和优化？

排查：java -jar app.jar --debug 打印自动配置报告，看哪些AutoConfiguration被加载了。

六大优化手段：
1. 排除不需要的自动配置：@SpringBootApplication(exclude={DataSourceAutoConfiguration.class})
2. 延迟初始化：spring.main.lazy-initialization=true（Bean首次使用时才创建）
3. 减少组件扫描范围：@ComponentScan("com.xx.core")
4. Spring Boot 3.x AOT编译：提前编译减少启动时间
5. JVM参数：-XX:+TieredCompilation -XX:TieredStopAtLevel=1 关闭C2编译，启动快
6. 升级到Spring Boot 3.4+：支持CDS(Class Data Sharing)

Spring Boot 3.x + Virtual Threads启动可以秒级。

Q10：线上出现内存泄漏，怎么在不重启的情况下止血？

紧急止损三步：
1. 限流保护：在Nginx/Gateway层对该接口限流，防止更多请求进来触发泄漏
2. 重启前分析：jmap -dump:live,file=heap.hprof PID 保留现场，然后重启
3. 离线分析dump：用MAT/JProfiler打开dump，看Dominator Tree——哪个对象占用最多内存→追溯GC Root→定位泄漏代码

常见泄漏源：
- ThreadLocal没remove（线程池复用线程，ThreadLocal值一直存在）
- static Map不断put不remove
- 打开流没close（FileInputStream/Socket）
- 监听器/回调注册后没注销
- 数据库连接池泄漏（没归还）

面试必背："先限流止损→dump现场→重启恢复→MAT分析根因。绝对不能一直重启——每次重启dump就丢了，永远找不到原因。"
