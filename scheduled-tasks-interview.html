Q1：@Scheduled 注解的原理是什么？怎么开启定时任务支持？

三步激活：
1. 启动类加 @EnableScheduling → Spring自动扫描@Scheduled
2. 方法上标注 @Scheduled(fixedRate=5000)
3. Spring容器启动后自动注册到 TaskScheduler 线程池

原理：@EnableScheduling 导入了 SchedulingConfiguration，注册一个 ScheduledAnnotationBeanPostProcessor——它在Bean初始化后扫描所有@Scheduled方法，包装成Runnable提交到TaskScheduler。

默认线程池大小是1！所有@Scheduled任务串行执行。生产环境必须自定义线程池。

Q2：fixedRate、fixedDelay、cron 三者的区别？

直观理解：
- fixedRate=5000：每5秒触发一次，不管上次是否执行完
- fixedDelay=5000：上次执行完后再等5秒才触发下一次
- cron="0 0 3 * * ?"：每天凌晨3点执行

选哪个？
统计类任务→fixedRate（定时采样）
依赖上次结果的任务→fixedDelay（批量处理）
精确到时刻→cron（日报/月报）

// fixedRate vs fixedDelay 的区别演示
@Scheduled(fixedRate = 5000)   // 每5秒启动一次（可能并发）
@Scheduled(fixedDelay = 5000)  // 上次结束+5秒后才启动（不会并发）
@Scheduled(cron = "0 0 8 * * MON")  // 每周一早8点

Q3：Cron表达式怎么写？常见6位和7位的区别？

Spring（6位）：秒 分 时 日 月 星期
Quartz（7位）：秒 分 时 日 月 星期 年（可选）

常用模板：
0 0/5 * * * ?         每5分钟
0 0 2 * * ?           每天凌晨2点
0 0 9-17 * * MON-FRI  周一到周五朝九晚五整点
0 30 10 1 * ?         每月1号10:30
0 0 0 L * ?           每月最后一天午夜
特殊字符：? (不指定) / L(最后) / W(工作日) / #(第几个)。面试说清楚秒分时日月的顺序就行。

Q4：Spring Boot定时任务的线程池怎么自定义？

默认问题：线程池大小=1，所有任务串行。一个任务堵了，后面的全排队。

解决方案： 
@Configuration
public class ScheduleConfig implements SchedulingConfigurer {
    @Override
    public void configureTasks(ScheduledTaskRegistrar registrar) {
        registrar.setScheduler(
            Executors.newScheduledThreadPool(10) // 核心10线程
        );
    }
}
生产推荐：用 ThreadPoolTaskScheduler（Spring包装），比Executors多了优雅关闭和异常处理。 
ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
scheduler.setPoolSize(10);
scheduler.setThreadNamePrefix("task-");
scheduler.setAwaitTerminationSeconds(60);
scheduler.setWaitForTasksToCompleteOnShutdown(true);

Q5：Quartz和XXL-JOB有什么区别？什么时候用哪个？

核心对比：
| 维度 | Quartz | XXL-JOB | |------|--------|---------| | 部署 | 内嵌应用，无独立控制台 | 独立调度中心+执行器 | | 可视化 | 没有（除非自建UI） | 自带的Web管理界面 | | 分布式 | 需额外配置 | 原生支持分片/故障转移 | | 动态修改 | 不行 | 可以在线改Cron，无需重启 | | 学习曲线 | 简单 | 中等 |
选型建议：单体项目用Quartz内嵌够用；微服务/需动态改任务的→XXL-JOB。面试时提一句：秒杀系统的库存预热定时任务就是XXL-JOB做的。

Q6：分布式环境下定时任务怎么保证不重复执行？

问题：三台机器都有同一任务，同时触发→重复处理数据。

三种解决方式：
1. Redis分布式锁：执行前 setIfAbsent(taskKey, 1, 60, TimeUnit.SECONDS)，拿到锁才执行
2. XXL-JOB分片广播：调度中心统一派活，不重复
3. 数据库乐观锁：UPDATE任务状态 version+1，竞争失败的跳过

面试加分："秒杀系统的预热任务就是通过Redis分布式锁保证多节点只有一台执行预热逻辑。"

Q7：定时任务的幂等性怎么保障？

为什么需要：任务可能超时重试或手动重跑→同一条数据被处理两次。

四种策略：
1. 业务唯一键：insert前检查唯一键是否已存在
2. 状态机：FROM pending TO processing；processing状态下跳过
3. Redis标记：处理完的数据ID存Redis，下次跳过
4. 乐观锁版本号：UPDATE SET status='done', version=version+1 WHERE version=旧值

实战示例： 
// 幂等处理：只有状态为PENDING的数据才处理
int updated = jdbc.update("UPDATE orders SET status='PROCESSING' WHERE id=? AND status='PENDING'", id);
if (updated == 0) return; // 已被处理，跳过

Q8：XXL-JOB的"分片广播"是什么？什么场景用？

概念：调度中心把任务拆成N份，同时发给N台执行器各自处理一部分。

原理：执行器拿到分片参数（shardIndex=当前是第几片，shardTotal=总共几片）→用 `id % shardTotal == shardIndex` 过滤自己要处理的数据。

典型场景：100万条订单需要在凌晨更新→10台机器各处理10万条，10倍加速。

代码示例： 
@XxlJob("batchUpdateOrders")
public void batchUpdate(int shardIndex, int shardTotal) {
    List orders = orderMapper.selectByShard(shardIndex, shardTotal);
    orders.forEach(this::processOrder);
}

Q9：定时任务执行超时了怎么办？怎么优雅停止一个正在跑的任务？

超时控制： 
Future future = taskScheduler.submit(task);
try {
    future.get(30, TimeUnit.SECONDS); // 最多等30秒
} catch (TimeoutException e) {
    future.cancel(true); // 超时→打断
    log.warn("任务超时，已中断");
}
优雅停止：用 Thread.interrupt() 发送中断信号，任务内部检查 Thread.currentThread().isInterrupted() 做收尾工作。

生产最佳实践：XXL-JOB自带任务超时kill + 失败重试 + 告警邮件。直接用就好。

Q10：ElasticJob和XXL-JOB怎么选？

ElasticJob：当当开源，基于ZooKeeper，偏技术驱动。分片策略丰富（平均分配/哈希/自定义），适合大规模数据处理。
XXL-JOB：许雪里开发，国产最火。有管理后台、在线编辑、报表看板，偏运维友好。

一句话选型：运维/产品经理也要能看→XXL-JOB；纯粹技术架构选→ElasticJob。面试被问就说"我用过XXL-JOB，有Web管理界面，对运维友好"。
