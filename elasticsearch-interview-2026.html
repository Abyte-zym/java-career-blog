🔍 Elasticsearch面试10题

2026 Java后端面试必备 · 覆盖倒排索引/集群/查询优化/脑裂

Q1: Elasticsearch的倒排索引是什么？

核心数据结构。传统索引：文档→词；倒排索引：词→文档列表。
每个Term维护Posting List（文档ID列表+词频+偏移量）。BKD Tree处理数值和地理坐标。
面试亮点：这就是ES搜索快的原因——O(1)定位到包含某个词的所有文档，不像MySQL要用LIKE全表扫。

Q2: ES的分词器（Analyzer）有哪些？怎么选？

standard：默认，按词边界切分+小写+停用词
ik_smart：最粗粒度，如"中华人民共和国国歌"→"中华人民共和国/国歌"
ik_max_word：最细粒度，如"中华人民共和国国歌"→"中华/人民/共和国/国歌"（召回更高）
pinyin：拼音分词，支持"zhangyanming"→"张晏铭"
选型原则：搜索用ik_max_word（召回高），索引用ik_smart（索引小）。混合策略——索引时smart+搜索时max_word。

Q3: ES集群架构：Master、Data、Ingest节点各干什么？

Master节点：管理集群（创建/删除索引、分配分片、维护元数据），不存数据。生产环境最少3个（脑裂防御）。
Data节点：存储数据+执行CRUD+搜索聚合。数据量大时拆分为hot/warm/cold分层。
Ingest节点：预处理管道（Grok解析日志、GeoIP解析IP、脚本转换）。ELK架构中Logstash的替代方案。
Coordinating节点：路由请求+合并结果，不存数据不做预处理。每个节点默认都是Coordinating角色。

Q4: 什么是分片（Shard）和副本（Replica）？

Primary Shard（主分片）：索引创建时确定，后期不可改。数据水平切分——每个分片是独立的Lucene实例。
Replica Shard（副本分片）：主分片的完整拷贝，数量可动态调整。作用：①故障转移（主分片挂了副本晋升）②提升查询吞吐（副本可分担读请求）。
公式：数据总量 / 单个分片建议≤50GB = 主分片数量。如500GB数据→10个主分片。
默认：1主+1副本=每个索引2个分片，各存不同节点。

Q5: ES写入流程（refresh / flush / translog）

①文档到达 → 写入内存Buffer + 同时写入Translog（WAL预写日志，防宕机）
②每隔1s（refresh_interval）→ 内存Buffer刷成Segment（此时文档可被搜索 = 近实时）
③Translog积累到512MB或30分钟 → Flush：Segment fsync到磁盘 + 清空旧Translog
# 调优：写入密集型场景增大refresh_interval PUT /my_index/_settings { "refresh_interval": "30s" }
关键：refresh保证近实时搜索，translog保证数据不丢。

Q6: ES查询为什么快？和MySQL对比

| 维度 | MySQL | Elasticsearch | | 核心 | B+树索引 | 倒排索引 | | 全文搜索 | LIKE '%xxx%'全表扫 | O(Term数量) | | 排序 | Filesort | 字段doc_values列式存储 | | 聚合 | 临时表 | 每个Segment独立聚合→合并 | | 相关性评分 | 无 | TF-IDF/BM25 | | 分布性 | 单机/分库分表 | 天生分布式 | ES快的原因：①倒排索引定位文档O(1) ②Segment不可变→OS Page Cache缓存命中率极高 ③Scatter-Gather并行查询。

Q7: ES脑裂（Split-Brain）怎么解决？

现象：网络分区导致集群分裂成两个子集群，各自选举Master→数据写入冲突→数据丢失。
解决：
① discovery.seed_hosts：指定候选Master节点列表
② cluster.initial_master_nodes：首次启动时指定参与选举的节点
③ minimum_master_nodes（7.x后自动）：N/2+1（过半原则），如3节点需至少2个认可才能当选
④ 避免对等Master：3节点时不要设2个master-eligible，单数+过半=稳定。

Q8: ES数据一致性怎么保证？

写入一致性（consistency）：
- one：主分片写入成功即可（默认）
- all：所有副本写入成功才返回
- quorum：多数分片写入成功
ES Near-Realtime特性：refresh_interval(1s)内的文档不可搜索——是近实时不是实时。如果需要强一致性，用`refresh=wait_for`参数。
版本控制：每个文档有_version字段，乐观锁并发控制——内部用_seq_no和_primary_term替代。

Q9: ES深度分页为什么慢？怎么优化？

FROM+SIZE的问题：ES要从每个分片取TOP(FROM+SIZE)条→汇总排序→取TOP SIZE条。如FROM=10000,SIZE=10 → 每个分片取10010条，5个分片=50050条→协调节点排序→取10条。内存爆炸。
# ❌ 深度分页 - 最多支持10000条（index.max_result_window） GET /posts/_search { "from": 10000, "size": 10 }
优化方案：
① Search After：用上一页最后一条的排序值做游标，走实时数据
② Scroll：生成快照，遍历大量数据（适合导出，不适合实时分页）
③ 业务层限制：只允许翻前100页

Q10: ES调优——你做过哪些优化？

硬件层：SSD硬盘（ES最吃IO）+ 64GB内存（50%给ES Heap，50%给OS Page Cache）
索引层：减少字段（`_source`按需include）、关闭不需要的`_all`、用`keyword`替代`text`做精确匹配
查询层：`filter`代替`query`（filter走缓存不计算分数）、用`routing`路由到指定分片、避免`wildcard`前导通配符
写入层：批量写入减少refresh次数、增大`translog flush_size`、写前关闭refresh写完再开
监控：`_cat/thread_pool`查队列积压、`_nodes/stats`查GC频率——Full GC超过每5秒一次就要调Heap大小。
