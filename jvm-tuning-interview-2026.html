Q1: 线上Java应用CPU飙升到100%，你怎么排查？
三步定位法：
① top -Hp PID — 找到CPU最高的线程ID
② printf '%x\n' TID — 转16进制
③ jstack PID | grep 0xTID -A 30 — 看这个线程在干什么

进阶：用 arthas 的 thread -n 3 直接看Top3繁忙线程，或 dashboard 实时监控。
常见原因：死循环、频繁GC、正则回溯、HashMap死链（JDK7）。
Q2: 内存泄漏怎么排查？
标准流程：
① jmap -histo:live PID | head -20 — 看哪些对象实例最多
② jmap -dump:live,format=b,file=heap.hprof PID — dump堆快照
③ 用 MAT / VisualVM 打开hprof，分析 Dominator Tree（支配树）

Arthas法： heapdump --live /tmp/dump.hprof 然后下载分析。
面试加分：提到 ThreadLocal未清理、静态集合持续增长、连接池未关闭 是Top3泄漏源。
Q3: GC频繁怎么办？如何看GC日志？
开启GC日志： 
-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags
分析工具： GCeasy（在线）、GCViewer
GC频繁原因+解决：
• Young GC频繁 → 年轻代太小，调大 -Xmn
• Full GC频繁 → 老年代满、元空间不足、System.gc() 调用
• 解决：调大堆、调GC算法（G1/ZGC）、查泄漏
G1关键参数： -XX:MaxGCPauseMillis=200 设置目标停顿
Q4: OOM有哪些类型？分别怎么解决？
| 类型 | 报错信息 | 常见原因 | 解决 | |------|---------|---------|------| | Java堆 | Java heap space | 大对象/内存泄漏 | -Xmx调大/排查泄漏 | | 元空间 | Metaspace | 动态类加载过多 | -XX:MaxMetaspaceSize | | 直接内存 | Direct buffer | NIO未释放 | -XX:MaxDirectMemorySize | | 栈溢出 | StackOverflow | 递归过深 | 检查递归终止条件 | | GC开销 | GC overhead limit | GC耗时>98% | 排查内存泄漏 |
Q5: 说一下G1和ZGC的区别？什么时候选ZGC？
| 特性 | G1 | ZGC | |------|----|-----| | 堆大小 | 4-64GB | 百GB~TB级 | | 停顿时间 | 200ms内 | <1ms（亚毫秒） | | 内存开销 | 10-20% | 使用染色指针 | | JDK版本 | JDK7+ | JDK15+（生产） | | 适用场景 | 通用 | 超低延迟、超大堆 | 选ZGC： 要求RT<1ms的金融/交易系统，或堆>=32GB时G1停顿不可接受。
ZGC核心： 染色指针（Colored Pointer）+ 读屏障（Load Barrier），并发执行几乎所有GC阶段。
Q6: 怎么用Arthas诊断线上问题？
最常用的5个命令：
① dashboard — 实时看线程/内存/GC
② thread -b — 找死锁（或 thread -n 3 看最忙线程）
③ jad com.xx.Service — 反编译线上代码（确认版本）
④ watch com.xx.Service method '{params,returnObj,throwExp}' -x 3 — 观测方法入参/返回值/异常
⑤ trace com.xx.Service method -n 5 — 追踪方法调用链和耗时
面试亮点： 提到"不重启应用就能诊断"是Arthas最大价值。
Q7: 堆Dump分析，MAT里面看什么？
MAT分析步骤：
① 打开 Leak Suspects Report — 自动识别疑似泄漏
② 看 Dominator Tree — 按占用内存排序，找到最大的对象
③ 右键 → Path to GC Roots → 看谁在引用它
④ Histogram — 统计类实例数和总大小

典型泄漏特征：
• HashMap$Node 数量异常多 → 静态Map未清理
• ThreadLocal 关联的Value存在 → Web应用中线程复用导致
• Connection 对象未关闭 → 连接池泄漏
Q8: 线上服务响应变慢，不是CPU也不是内存问题，怎么排查？
排除CPU/内存后，重点查：
① netstat -antp | wc -l — 连接数是否过多
② iostat -x 1 — 磁盘IO是否瓶颈
③ strace -p PID -c — 系统调用耗时分布
④ Arthas trace 命令追踪每个方法耗时
⑤ 查看数据库慢查询日志（可能DB端出问题）
⑥ 检查下游服务超时（RocketMQ/Kafka消费延迟）
面试重点： 展现"全链路排查思维"，不只盯着Java进程。
Q9: 生产环境怎么优雅地做JVM参数调优？
标准流程：
① 基线： 先用默认参数跑，记录GC日志+堆使用率
② 压测： 模拟真实流量，观察GC频率和停顿
③ 调参： 根据GC日志逐步调整 -Xms=Xmx、-Xmn、GC算法
④ 灰度： 先在1-2台机器上验证，不要全量改
⑤ 监控： 上线后持续观察Prometheus + Grafana面板

推荐生产参数模板： 
-Xms4g -Xmx4g -Xmn1.5g
-XX:+UseG1GC -XX:MaxGCPauseMillis=200
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/heap.hprof
Q10: 你做过哪些JVM调优？踩过什么坑？
面试话术（结合秒杀项目）：
"秒杀项目压测时发现QPS到3000后频繁Full GC，排查发现：
① 秒杀返回结果直接用 new String(byte[]) 创建了大量临时字符串
② 改用StringBuilder拼接后，Young GC频率从每5秒降到每30秒
③ 另外发现Redis的Lettuce连接池没限制大小，改后堆使用率从85%降到60%。
关键经验： 小对象频繁创建是Young GC的元凶，先看GC日志再动参数，别上来就调堆大小。"
