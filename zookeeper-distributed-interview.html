Q1：ZooKeeper是什么？解决了什么问题？

一句话：分布式协调服务。核心能力是统一命名、配置管理、集群管理、分布式锁。

解决的实际问题：
• Dubbo用ZK做服务注册发现——Provider上线注册临时节点，Consumer watch节点变化
• 分布式锁——多个服务抢同一商品库存时，ZK临时顺序节点保证只有一个拿到锁
• 配置中心——所有服务watch同一个配置节点，改一处所有服务自动感知
• 选主——多个备份节点通过ZK选举出一个Master

面试时提：Kafka早期用ZK管理Broker和Topic，新版已去ZK化。但Dubbo/Hadoop/微服务治理中ZK仍是标配。

Q2：ZAB协议是什么？和Raft有什么区别？

ZAB（ZooKeeper Atomic Broadcast）：ZK自研的一致性协议，设计目标是崩溃恢复+原子广播。

两种模式：
1. 崩溃恢复模式：Leader挂了→重新选举→新Leader同步数据给Follower
2. 消息广播模式：Leader接收写请求→转发Proposal给所有Follower→过半ACK→提交

和Raft的核心区别：
| ZAB | Raft |
| 专门为ZK设计 | 通用共识算法 |
| Leader广播是FIFO顺序 | 日志复制严格有序 |
| 新Leader直接同步最新数据 | 新Leader从最新日志开始 |
| 实现较复杂 | 设计更简洁易懂 |

面试时只说一句话："ZAB是ZK的专用共识协议，两阶段提交+过半确认，和Raft思想类似但实现不同。"

Q3：ZK的选举机制是怎样的？

Fast Leader Election（快速选举算法）：
每个节点投票时比较（ZXID大者优先 → 相同则myid大者优先）。

选举流程：
1. 每个节点给自己投票(myid, ZXID)
2. 广播自己的投票给集群其他节点
3. 收到别人的投票后，比较：谁的ZXID大选谁；ZXID相同比myid
4. 改投更强的节点，再次广播
5. 当某个节点获得超过半数的投票→成为Leader
6. Leader广播LEADING状态，Follower广播FOLLOWING

关键点：ZXID是事务ID，事务越多的节点数据越新→优先当Leader保证数据不丢。

Q4：ZK如何实现分布式锁？

核心方案：临时顺序节点 + Watch机制

加锁流程：
1. 所有客户端在 /lock 下创建临时顺序节点，如 /lock/seq-0000000001
2. 检查自己是不是序号最小的节点
3. 是→获得锁，执行业务
4. 不是→Watch前一个节点（比自己序号小的最大那个）
5. 前一个节点释放（客户端断开或删除）→收到通知→重新检查→自己变最小→获得锁

优点：FIFO公平锁，不会惊群效应——只唤醒下一个等待者。
和Redis锁对比：ZK锁强一致性但性能较低；Redis锁性能高但可能丢锁（主从切换）。秒杀场景用Redis，支付场景用ZK。

Q5：ZK节点的4种类型？各自什么场景用？

| 类型 | 特点 | 典型场景 | |------|------|---------| | 持久节点 | 创建后一直存在，手动删除 | 配置信息 | | 持久顺序节点 | 持久+自动编号 | 全局唯一ID | | 临时节点 | 客户端断开自动删除 | 服务注册/心跳 | | 临时顺序节点 | 临时+自动编号 | 分布式锁/选主 |
面试亮点：临时节点的生命周期绑定客户端Session——这也是ZK能自动感知服务下线的原理。Dubbo的Provider挂了→临时节点消失→Consumer的watch触发→更新本地服务列表。

Q6：Watcher机制有什么特点？

三个核心特点：
1. 一次性触发：Watcher触发后失效，需要重新注册
2. 串行执行：所有Watcher回调串行执行（一个处理完才到下一个）
3. 轻量级：Watcher传递的是事件类型+节点路径，不含数据

为什么是一次性的？如果服务端持续推送会浪费带宽。客户端收到通知后自己去getData()拿最新数据+重新注册Watcher。

常见陷阱：Watcher回调里不能做耗时操作——因为是串行的，会阻塞其他Watcher。

Q7：ZK是CP还是AP？为什么？

答案：CP（强一致性+分区容错）。

证明过程：
• C（一致性）：Leader挂了期间不对外提供服务→牺牲可用性保一致性
• P（分区容错）：过半节点存活就能选举和提交→容忍网络分区
• 不是A（可用性）：选举期间整个集群不可用（通常几秒到30秒）

对比Eureka：Eureka是AP——宁可返回旧数据也不拒绝服务。适合服务发现的一致性要求不高的场景。
对比Nacos：Nacos同时支持AP和CP模式，注册中心用AP（优先可用），配置中心用CP（要求一致性）。

Q8：ZK集群为什么是奇数台？

核心原因：过半同意机制（Quorum）。

3台 vs 4台对比：
• 3台集群：容忍1台故障（2>3/2）
• 4台集群：也容忍1台故障（3>4/2）
→ 多花一台机器的钱，容错能力没增加。

5台集群：容忍2台故障。
结论：可用节点数 = 总节点数 - 故障数；但提交要求 故障数 < 总节点数/2。所以奇数台机器性价比最高。

面试一句话："3台和4台都只能挂1台，多一台白花钱。所以生产环境一般用3台或5台。"

Q9：ZK的Session超时怎么处理？临时节点什么时候删除？

Session生命周期：
1. 客户端连接ZK→创建Session（分配SessionID+Timeout）
2. 客户端定期发心跳（Ping）
3. 服务端在sessionTimeout时间内没收到心跳→标记Session为Expired
4. Session过期→该客户端创建的所有临时节点全部删除
5. 其他Watch这些节点的客户端收到通知

关键参数：
• sessionTimeout：服务端允许的最大超时时间，客户端连接时声明
• tickTime：ZK基本时间单位，默认2000ms
• 最小超时=2×tickTime，最大超时=20×tickTime

面试陷阱：临时节点是客户端断开连接后才删除，不是心跳断了立刻删。需要等sessionTimeout到期。

Q10：ZK不适合什么场景？

ZK的局限性：
1. 不适合大数据存储：单个节点数据量建议<1MB，不是数据库
2. 不适合高并发写：所有写都要Leader→事务→广播，写QPS几百
3. 不适合实时性要求高的场景：选举期间不可用（几秒~30秒）
4. 不适合做消息队列：Watcher一次性且无持久化消费能力
5. 不适合跨机房部署：网络延迟大会严重影响Leader选举和ZAB提交

替代方案：
• 配置中心→Nacos/Apollo（更轻量）
• 分布式锁→Redis Redisson（更高性能）
• 选主→Etcd（K8s生态标配）
• 服务发现→Nacos/Eureka（专为此设计）
