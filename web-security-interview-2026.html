Q1：CSRF攻击原理？Spring Security怎么防护？

原理：攻击者诱导已登录用户点击恶意链接，利用浏览器自动携带Cookie的机制，冒充用户发起请求（如转账、改密码）。发生在第三方站点，浏览器同源策略不阻止跨站请求的发送。

Spring Security防护：

// Spring Security默认开启CSRF防护
http.csrf().csrfTokenRepository(
    CookieCsrfTokenRepository.withHttpOnlyFalse()
);
// Token机制：服务端生成随机Token → 前端每个请求携带 → 后端校验
// SameSite Cookie：设置Cookie的SameSite=Strict/Lax

面试口诀："CSRF利用Cookie自动携带 → Spring Security默认Token防护 → SameSite双重保险"

Q2：XSS攻击类型有哪些？后端如何防御？

三种类型：

  • 存储型：恶意脚本存入数据库，所有人访问页面时执行（危害最大）
  • 反射型：脚本写在URL参数中，服务端反射回页面时执行
  • DOM型：纯前端漏洞，js从URL取参数直接写入DOM

后端防御方案：

// 1. 输出编码（核心）
String safe = HtmlUtils.htmlEscape(userInput);
// 2. 输入校验白名单
@Pattern(regexp = "^[a-zA-Z0-9]+$")
private String username;
// 3. CSP响应头
response.setHeader("Content-Security-Policy", 
    "default-src 'self'; script-src 'self'");
// 4. HttpOnly Cookie（防XSS窃取Cookie）
cookie.setHttpOnly(true);

Q3：SQL注入原理及MyBatis如何防止？

原理：攻击者在输入字段拼接SQL片段（如 ' OR '1'='1），改变原始SQL语义，获取未授权数据或执行破坏性操作。

// ❌ 危险：${} 直接拼接，存在注入风险
@Select("SELECT * FROM users WHERE name = '${name}'")

// ✅ 安全：#{} 预编译，参数化查询
@Select("SELECT * FROM users WHERE name = #{name}")

// MyBatis底层：PreparedStatement → 参数与SQL分离 → 无法改变SQL结构

Q5：HTTPS加密原理？SSL/TLS握手过程用3句话概括

一句话版本：客户端验证服务器证书（非对称加密）→ 双方协商生成临时对称密钥 → 之后全部用对称密钥加密通信。

三句话概括：

  1. ClientHello → ServerHello + 证书（含公钥）→ 客户端用CA根证书验证服务器身份
  2. 客户端生成pre-master secret → 用服务器公钥加密发送 → 双方各自推导出对称会话密钥
  3. Finished消息互相确认 → 后续全部对称加密（AES）通信

     → Controller

Q7：什么是OAuth2.0？授权码模式流程？

核心角色：Resource Owner（用户）→ Client（第三方应用）→ Authorization Server（授权服务器）→ Resource Server（资源服务器）。

授权码模式（最安全）：

  1. 用户点击"微信登录" → 跳转授权页面 → 用户同意
  2. 授权服务器返回 authorization_code 给客户端（通过浏览器重定向）
  3. 客户端用code + client_secret 换 access_token（后端调用，code不暴露给前端）
  4. 客户端用access_token访问资源服务器

为什么最安全：token不经过浏览器，code只能用一次且在服务器间交换。

Q8：Cookie的Secure、HttpOnly、SameSite属性含义？

// 三个安全属性组合使用
Cookie cookie = new Cookie("token", jwt);
cookie.setHttpOnly(true);   // JS无法读取 → 防XSS窃取
cookie.setSecure(true);     // 仅HTTPS传输 → 防中间人截获
cookie.setAttribute("SameSite", "Strict"); // 仅同站请求携带 → 防CSRF
属性作用防御
HttpOnly禁止JS访问XSS
Secure仅HTTPS中间人
SameSite限制跨站发送CSRF

Q9：密码存储最佳实践？BCrypt vs MD5？

// ❌ 绝对禁止
String hash = MD5(password);              // 彩虹表秒破
String hash = SHA256(password);           // GPU暴力破解

// ✅ 正确做法
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
BCryptPasswordEncoder encoder = new BCryptPasswordEncoder();
String hash = encoder.encode(password);   // $2a$10$...每次结果不同

// BCrypt核心优势：
// 1. 内置盐值（自动生成随机salt，存于hash中）
// 2. 计算慢（cost factor控制迭代次数，默认10=2^10次）
// 3. 每次结果不同（相同密码两次encode结果不同）

Q10：什么是CORS？Spring Boot如何配置跨域？

原理：浏览器的同源策略默认禁止跨域请求。CORS通过HTTP头让服务器声明允许哪些源访问。

// 方式1：@CrossOrigin注解
@CrossOrigin(origins = "https://example.com")
@RestController
public class MyController { }

// 方式2：全局配置（推荐）
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://example.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowedHeaders("*")
            .allowCredentials(true)
            .maxAge(3600);
    }
}
// 关键响应头：Access-Control-Allow-Origin
