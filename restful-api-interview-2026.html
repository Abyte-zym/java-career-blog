Q1: GET和POST的区别？为什么GET是幂等的？

🔑 标准答案：

GET用于获取资源，POST用于创建资源。关键区别：

✅ GET幂等 — 调用100次结果相同，不会改变服务器状态

✅ POST非幂等 — 每次调用创建新资源，调用N次产生N条记录

✅ GET参数在URL中（长度限制），POST参数在请求体中（无长度限制）

✅ GET可被缓存、收藏、分享，POST不行

💡 加分回答："严格RESTful中，GET是安全且幂等的，PUT是幂等（更新同一资源），POST是不幂等（创建新资源），DELETE是幂等（删除后再次删除结果相同——资源不存在）。"

Q2: PUT和PATCH的区别？各自应用场景？

🔑 标准答案：

✅ PUT — 全量更新。客户端传入完整资源对象，替换整个资源

✅ PATCH — 部分更新。客户端只传入要修改的字段

✅ PUT是幂等的（重复更新结果相同），PATCH不一定幂等

// PUT：替换整个用户
PUT /users/1 {"name":"张三","email":"zhang@test.com","age":22}

// PATCH：只改年龄
PATCH /users/1 {"age":23}

Q3: 常见HTTP状态码及其含义？

🔑 标准答案（按5类记忆）：

✅ 2xx 成功：200 OK / 201 Created(POST成功) / 204 No Content(DELETE成功)

✅ 3xx 重定向：301 永久移动 / 302 临时移动 / 304 未修改(缓存生效)

✅ 4xx 客户端错误：400 参数错误 / 401 未认证 / 403 无权限 / 404 资源不存在 / 405 方法不允许 / 429 请求过多

✅ 5xx 服务端错误：500 服务器内部错误 / 502 网关错误 / 503 服务不可用

💡 面试追问："201和200区别？" → 201在POST创建资源后返回，且应包含Location头指向新资源URL

Q4: Spring Boot中@RestController和@Controller区别？

🔑 标准答案：

✅ @RestController = @Controller + @ResponseBody

✅ @Controller返回视图名（通过ViewResolver解析为HTML页面）

✅ @RestController每个方法返回值直接序列化为JSON/XML写入HTTP响应体

✅ 前后端分离项目100%用@RestController

@RestController  // ← 每个方法return都变成JSON
@RequestMapping("/api/users")
public class UserController {
    @GetMapping
    public List<User> list() {
        return userService.findAll();  // → [{"id":1,"name":"张三"},...]
    }
}

Q5: RESTful API如何做版本管理？

🔑 标准答案（4种方式，推荐方案1）：

✅ 方案1（推荐）：URL路径版本 — /api/v1/users /api/v2/users

✅ 方案2：请求头版本 — Accept: application/vnd.myapp.v1+json

✅ 方案3：参数版本 — /api/users?version=1

✅ 方案4：域名版本 — v1.api.example.com

💡 实际项目做法：URL路径版本最直观、最容易测试、Nginx路由最方便。Spring Boot中新建Controller，路径前缀不同即可。

Q6: 什么是HATEOAS？实际用过吗？

🔑 标准答案：

✅ HATEOAS = 超媒体作为应用状态引擎

✅ API响应中不仅返回数据，还返回下一步可执行的操作链接

✅ 例如：查询订单后自动返回"取消订单"/"确认收货"等操作URL

// 普通REST返回：
{"id":1, "status":"paid"}

// HATEOAS返回：
{
  "id":1, "status":"paid",
  "_links": {
    "cancel": {"href":"/orders/1/cancel"},
    "refund": {"href":"/orders/1/refund"}
  }
}

💡 诚实回答："了解原理，Spring HATEOAS框架有点重。实际项目用Swagger文档描述API即可，HATEOAS的理想不错但落地不多。"

Q7: API接口如何防止重复提交？

🔑 标准答案（3层防御）：

✅ 第1层：前端防抖 — 按钮点击后置灰+loading，1秒内不可重复点击

✅ 第2层：Token机制 — 打开表单时后端下发唯一token，提交时带上，后端用Redis setIfAbsent验证+删除

✅ 第3层：业务幂等键 — 订单号/业务流水号在数据库唯一索引，重复插入直接忽略

// 防重Token核心代码
String token = request.getHeader("X-Idempotent-Token");
Boolean locked = redis.setIfAbsent("idem:"+token, "1", 5, TimeUnit.MINUTES);
if (!locked) throw new BizException("请勿重复提交");

Q8: 如何处理API的跨域（CORS）问题？

🔑 标准答案：

✅ CORS = 浏览器同源策略限制，跨域请求浏览器先发OPTIONS预检请求

✅ Spring Boot解决方案：

@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://your-frontend.com")
            .allowedMethods("GET","POST","PUT","DELETE")
            .allowCredentials(true);
    }
}

✅ Nginx解决方案：add_header Access-Control-Allow-Origin *;（生产环境不用*，指定域名）

💡 面试追问："OPTIONS预检请求什么时候发？" → 非简单请求（Content-Type为application/json、自定义请求头）都会发预检

Q9: API接口返回的JSON格式应该怎么设计？

🔑 标准答案：

✅ 统一响应格式：

public class R<T> {
    private int code;     // 200成功 4xx客户端错 5xx服务端错
    private String msg;   // 可读的错误描述
    private T data;       // 实际数据
    private long timestamp;
    
    public static <T> R<T> ok(T data) { return new R<>(200,"success",data); }
    public static R<Void> error(int code, String msg) { return new R<>(code,msg,null); }
}

✅ 分页响应：data里包一层：{"total":100, "page":1, "size":20, "list":[...]}

✅ 前端统一拦截code，非200直接提示msg，不需要每个接口单独判断

Q10: RESTful API的安全措施有哪些？

🔑 标准答案：

✅ 认证：JWT Token（无状态）或OAuth2（第三方登录）

✅ 授权：角色/权限控制（ADMIN/用户只能操作自己的数据）

✅ 传输安全：HTTPS加密传输

✅ 输入校验：@Valid + 参数校验注解，防止SQL注入和XSS

✅ 限流：API网关/Sentinel限制单IP/单用户QPS

✅ 日志脱敏：密码/手机号不打印到日志

✅ 幂等控制：支付类接口必须有防重机制
