Q1：什么是CAP定理？为什么CAP只能三选二？
标准答案：
  • C（一致性）：所有节点同一时刻数据一致
  • A（可用性）：每个请求都能得到响应（无论成功失败）
  • P（分区容错）：网络分区时系统仍能工作
  • 网络分区不可避免，所以 P 必须选——实际是 CP 或 AP 的选择
  • CP：ZooKeeper（分区时停写，保证一致性）
  • AP：Eureka（分区时继续服务，允许短暂不一致）
// 面试加分句：
"CAP指导架构选型——金融场景选CP(Nacos选CP模式)，高可用场景选AP(Redis Cluster)"

Q4：分布式一致性协议 Raft 和 Paxos 的核心区别？
标准答案：
  • Paxos：论文提出，理论和实现复杂，允许乱序提交
  • Raft：强领导模型，Leader选举+日志复制，易懂
  • 核心流程：Leader选举（term递增）→ 日志复制（过半提交）
  • 应用：etcd、TiKV、Nacos（CP模式）
// 面试加分句：
"Raft本质是Paxos的工程化——用Leader强制顺序简化了Paxos的乱序提交
 Nacos CP模式基于Raft，AP模式基于Distro"
Q5：雪花算法（Snowflake）的ID结构是什么？时钟回拨怎么处理？
标准答案： 
Snowflake ID = 1bit(0) + 41bit(毫秒) + 10bit(机器ID) + 12bit(序列号)

每秒可生成 2^12 × 1000 = 409.6万个ID
时钟回拨处理：
  • 方案1：等待时钟追上（短回拨适用）
  • 方案2：备用机器ID（发生回拨自动切换）
  • 方案3：美团Leaf方案——号段模式，完全避免时钟问题
Q6：分布式ID生成有哪些方案？
UUID（无序影响索引）→ 雪花算法（时钟依赖）→ 美团Leaf号段（推荐）→ Redis自增（简单但不持久）
Q7：接口幂等性怎么保证？
4种方案：①唯一索引（数据库约束）②Redis setIfAbsent去重key ③状态机（待支付→已支付不可逆）④Token机制（前端获取token→提交时校验+删除）
秒杀项目用Redis + 唯一订单号双重保障。
Q8：限流算法有哪些？令牌桶 vs 漏桶区别？
漏桶：恒定速率流出，突发流量被平滑 → 适合保护下游
令牌桶：固定速率放令牌，允许突发 → 适合秒杀场景
滑动窗口：精确到毫秒级计数 → Sentinel默认

Q9：服务注册发现的原理是什么？
  • 服务提供者启动→向注册中心注册（Nacos/Eureka）
  • 注册中心维护健康检查（心跳/主动探测）
  • 消费者→拉取（Pull）+ 订阅（Push）服务列表
  • 服务下线→注册中心通知消费者剔除

Q10：如何保证分布式系统数据最终一致性？
核心手段：①消息队列异步（削峰+重试）②本地消息表（发MQ前先写表）③定时对账（兜底补偿）④人工介入（死信队列告警）
秒杀项目实践：Redis扣库存→MQ异步下单→订单表+消息表双写→定时扫描未完成消息→重试/告警。
