Q1: 什么是RAG？它的核心流程是什么？
RAG = Retrieval-Augmented Generation（检索增强生成）。核心流程三步：

① 文档预处理：上传PDF/Word→解析→分块（Chunk 500-1000字）→Embedding向量化→存入向量数据库（Redis/PGVector）
② 检索：用户提问→同样向量化→向量库Top-K相似度检索（通常K=3-5）
③ 生成：检索结果拼入Prompt→调用大模型→基于上下文生成回答

为什么用RAG？解决大模型的两大痛点：知识时效性（不依赖训练数据截止日期）和幻觉问题（回答有据可查）。
Q2: Spring AI是什么？和LangChain4j有什么区别？
Spring AI是Spring官方推出的AI集成框架（2026年2.0已GA），让Java开发者用Spring生态的方式接入大模型。

核心能力：
• ChatClient — 对话接口，支持流式输出
• EmbeddingModel — 文本向量化
• VectorStore — 向量数据库抽象（Redis/PGVector/Milvus）
• Tool Calling — Agent工具调用

vs LangChain4j：Spring AI 天然集成Spring Boot生态（@Bean注入、Actuator监控、@Conditional自动配置），对Spring团队更友好。LangChain4j社区更大、模型支持更广。
Q3: AI Agent的核心公式是什么？Agent怎么实现工具调用？
Agent = LLM + Planning（规划）+ Memory（记忆）+ Tools（工具）

工具调用流程：
① LLM分析用户意图→决定调用哪个工具
② 输出结构化指令（如{"tool":"search","args":{"query":"Java面试"}}）
③ 代码层解析指令→执行工具→拿结果
④ 结果反馈给LLM→LLM基于结果生成最终回答

关键：模型只做决策（选哪个工具、传什么参数），真正执行是你的代码。Spring AI通过@Tool注解注册工具方法。
Q4: Prompt工程的核心技巧有哪些？
五大技巧：
① 角色设定："你是一个资深Java面试官..." — 限定回答风格
② Few-shot示例：给2-3个问答范例，引导输出格式
③ 思维链（CoT）：加"让我们一步步思考" — 提升推理准确率40%+
④ 结构化输出：要求返回JSON/Markdown格式
⑤ 负面约束："不要编造信息，不知道就说不知道" — 防幻觉

2026年趋势：自动Prompt优化（DSPy）+ 多模态Prompt（图文混合）。
Q5: 向量数据库怎么选？Redis和Milvus的区别？
| 维度 | Redis Stack | Milvus | |------|------------|--------| | 定位 | 缓存+向量二合一 | 专用向量数据库 | | 性能 | 低延迟（内存） | 高吞吐（GPU加速） | | 规模 | 百万级向量 | 十亿级向量 | | 运维 | 轻量，已有Redis团队首选 | 需独立部署 | | 场景 | 中小规模RAG | 企业级语义搜索 |

我的项目选Redis：秒杀系统已用Redis做缓存，加向量功能无需引入新组件，架构简洁。
Q6: 怎么解决RAG的"上下文窗口"问题？
检索回来的文档片段总和可能超过大模型的Token上限。解决方案：

① 重排序（Re-ranking）：先用简单检索召回Top-10，再用Cross-encoder精排保留Top-3
② 切片策略优化：小chunk（256-512 token）+ overlap（10-20%）
③ 摘要压缩：检索后先用小模型压缩片段再送入大模型
④ Token计数防御：代码层面检查totalTokens > maxTokens时自动截断
Q7: 如何评估RAG系统的回答质量？
三个维度：
① 忠实度（Faithfulness）：回答是否基于检索的文档？→ 用NLI模型检测幻觉比例
② 相关性（Relevance）：回答是否切题？→ 人工打分或用GPT-4做裁判
③ 召回率（Recall）：检索到了多少相关文档？→ 对比关键词检索和向量检索的命中率

我的项目实践：每次RAG调用记录"检索文档数+生成耗时+用户反馈"三元组，持续监控质量。
Q8: Spring AI 2.0有哪些新特性？（2026必问）
① Tool Calling升级：@Tool注解简化工具注册，支持多工具并行调用
② Advisors链：类似Spring Interceptor的AOP机制，可插入日志/重试/缓存
③ 多模态支持：图片+文本混合输入（GPT-4o/Claude Vision）
④ ChatMemory：内置对话记忆管理（内存/JDBC/Redis实现）
⑤ 速率限制：内置Token桶限流，防止API超额
⑥ Spring Boot 4.x兼容：全面适配最新Spring生态
Q9: 项目中怎么避免大模型API调用超时和失败？
三层防护：
① 重试机制：指数退避重试（1s→2s→4s，最多3次），@Retryable注解
② 超时设置：连接超时5s + 读取超时30s，超过返回降级回答
③ 熔断降级：连续失败5次→熔断30秒→返回缓存答案或"服务繁忙"

监控：记录每次AI调用的耗时+成功/失败+Token消耗，接入Actuator端点。
