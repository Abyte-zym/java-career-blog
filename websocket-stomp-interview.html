Q1：WebSocket和HTTP有什么区别？什么时候用WebSocket？

核心区别：
HTTP：请求-响应模式，客户端问→服务端答→连接关闭。单向主动。
WebSocket：全双工持久连接，握手后双方随时发消息，不需重新建立连接。

用WebSocket的场景：聊天IM、股票行情、协同编辑、游戏对战、消息推送——任何需要服务端主动推数据给客户端的场景。

面试时一定提这句："WebSocket基于TCP，先通过HTTP Upgrade握手升级协议，之后在这个TCP连接上进行全双工通信。"

Q2：WebSocket握手过程是怎样的？

三步握手：
1. 客户端发起HTTP请求，带 Upgrade: websocket 和 Connection: Upgrade 头
2. 服务端返回 101 Switching Protocols
3. TCP连接升级为WebSocket，后续走WebSocket帧协议

关键Header：
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
服务端用 Sec-WebSocket-Key + 固定GUID做SHA1→Base64返回 Sec-WebSocket-Accept，防止缓存代理误缓存。

Q3：STOMP协议是什么？和原始WebSocket什么关系？

STOMP（Simple Text Oriented Messaging Protocol）：基于WebSocket之上的子协议，提供了"消息目的地"概念（类似MQ的topic/queue），让WebSocket消息有了路由能力。

对比：
原始WebSocket：你发什么字节流对方收什么，需要自己定义消息格式和路由逻辑。
STOMP：自动提供 /topic/xxx（广播）和 /queue/xxx（点对点）路由，Spring内置支持。

实际项目用STOMP的原因："不用自己造轮子——Spring的 @MessageMapping + SimpMessagingTemplate 直接搞定消息路由和推送。"

Q4：Spring怎么配置WebSocket+STOMP？

核心配置类： 
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {
    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws").setAllowedOriginPatterns("*").withSockJS();
    }
    @Override
    public void configureMessageBroker(MessageBrokerRegistry registry) {
        registry.enableSimpleBroker("/topic", "/queue");  // 订阅前缀
        registry.setApplicationDestinationPrefixes("/app"); // 发送前缀
    }
}
关键点：
- /ws 是WebSocket连接端点
- /topic 是广播前缀，/queue 是点对点前缀
- /app 是客户端发送消息的前缀，对应Controller里的 @MessageMapping("/chat")

Q5：WebSocket怎么保证心跳和断线重连？

三层保障：
1. WebSocket协议层心跳：Ping/Pong帧，浏览器自动处理
2. STOMP层心跳：客户端连接时设置 heartbeat.outgoing 和 heartbeat.incoming（毫秒）
3. 应用层重连：SockJS的 disconnect 事件触发重连逻辑

// 客户端STOMP重连
const socket = new SockJS('/ws');
const client = Stomp.over(socket);
client.connect({}, frame => {
    // 连接成功
}, error => {
    setTimeout(() => reconnect(), 3000); // 3秒后重连
});
服务端心跳超时检测：Spring默认10秒无心跳断开连接，可通过 setTaskScheduler 配置。

Q6：WebSocket vs HTTP长轮询 vs SSE，怎么选？

三兄弟对比：
方案原理适用场景
HTTP长轮询客户端hold住请求，有数据才返回兼容老浏览器、简单场景
SSE服务端→客户端单向推送股票行情、日志推送、AI流式输出
WebSocket全双工双向通信聊天IM、协同编辑、游戏

实习生加分回答："我的RAG项目里AI回答用的是SSE流式输出——因为只需要服务端推文字，不需要客户端发消息。秒杀系统的秒杀结果推送才适合WebSocket——需要双向交互。选型不只考虑技术，更考虑场景。"

Q7：WebSocket连接数太多怎么扩展？

两种思路：
1. 垂直扩展：提高单机连接数上限——Linux调 ulimit -n 文件描述符限制、Spring Boot调 server.tomcat.max-connections
2. 水平扩展：多节点+消息队列

水平扩展方案：
- 用Redis Pub/Sub做消息广播：同一用户连NodeA，消息到Redis→所有节点收到→推给对应客户端
- 或用RabbitMQ/Kafka：节点只负责WebSocket连接，消息分发走MQ

一句话解释："WebSocket是有状态连接，不能像HTTP一样无脑负载均衡。需要引入消息中间件实现跨节点消息路由。"

Q8：怎么统计当前在线WebSocket用户数？

方案：监听连接/断开事件 
@Component
public class WebSocketEventListener {
    private final AtomicInteger onlineCount = new AtomicInteger(0);
    
    @EventListener
    public void handleSessionConnected(SessionConnectEvent event) {
        onlineCount.incrementAndGet();
    }
    
    @EventListener
    public void handleSessionDisconnect(SessionDisconnectEvent event) {
        onlineCount.decrementAndGet();
    }
    
    public int getOnlineCount() { return onlineCount.get(); }
}
分布式场景：每个节点统计自己的连接数→定时上报Redis→汇总。用 REDIS INCR/DECR 配合节点标识做原子计数。

Q9：WebSocket消息如何保证不丢失？

三层保障：
1. 客户端ACK：STOMP支持 ack: 'client' 模式，客户端收到后回ACK，未ACK的消息重发
2. 服务端离线消息：用户不在线时消息存Redis/MQ，上线后拉取
3. 消息持久化：关键消息（如支付通知）先写DB，推送成功后标记已推送

实习生面试够用的说法："方案是客户端ACK + 离线消息队列 + 重试机制。消息先写Redis的List，推送成功删除，失败则保留供重试。"

Q10：Nginx怎么代理WebSocket？

关键配置： 
location /ws {
    proxy_pass http://websocket_backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 3600s;  # 长连接不超时
    proxy_set_header Host $host;
}
三个必加配置解释：
- proxy_http_version 1.1：WebSocket需要HTTP/1.1
- Upgrade 和 Connection "upgrade"：告诉Nginx这是WebSocket升级请求，不要当普通HTTP处理
- proxy_read_timeout 3600s：WebSocket是长连接，默认60s超时不够

不配这三个会怎样？WebSocket连上一会儿就断开——Nginx当成普通HTTP连接，超时后就关了。
