Q1：为什么需要分布式ID生成器？数据库自增ID有什么问题？

数据库自增ID在单机没问题，分库分表后有三个致命问题：
1. ID冲突：两个分库各自从1开始自增，主键重复。
2. 暴露业务量：自增ID让竞品能算出来你每天多少订单。
3. 迁移困难：数据迁移时ID冲突，合并后主键全乱。

所以需要全局唯一的分布式ID。

Q2：雪花算法（Snowflake）原理是什么？64位怎么划分？

Twitter开源的Snowflake，一个64位Long：
0 - 0000000000 0000000000 0000000000 0000000000 0 - 0000000000 - 000000000000
│   │                      │                          │            │
│   └── 41位时间戳          └── 10位机器ID              └── 12位序列号
└── 1位保留（始终0，保证ID为正）
41位时间戳=从自定义起始时间（如2020-01-01）的毫秒数，够用69年。
10位机器ID=1024台机器，分5位数据中心+5位工作节点。
12位序列号=同一毫秒内自增，每毫秒4096个ID。

Q3：雪花算法有什么缺点？时钟回拨怎么解决？

最大问题是时钟回拨：服务器时间被调回过去→可能生成重复ID。

四种解法：
1. 抛异常（最暴力的）：检测到回拨直接抛异常。
2. 等待追平：回拨幅度小（<5ms）就sleep等待追上。
3. 备用workerId：回拨时换一个新workerId，旧workerId记录最后一次时间。
4. 扩展位（百度UidGenerator方案）：借用未来时间，用sequence填满后自动切换。

面试多说一句：生产上用NTP时间同步服务+监控告警防止大幅回拨。

Q4：雪花算法和UUID各有什么优缺点？

维度雪花算法UUID
大小8字节（Long）16字节（128位）
趋势递增✅ 大致有序❌ 无序
数据库索引友好✅ B+Tree写友好❌ 随机IO，页分裂严重
依赖机器时钟无依赖
可读性纯数字字符串
分布式需配置workerId天然分布式
结论：订单ID用雪花算法（排序查询），文件ID用UUID（不冲突）。实习生知道怎么选就够了。

Q5：美团Leaf方案怎么工作的？

Leaf有两种模式：
1. Leaf-segment（号段模式）：从DB批量取一段ID（如1001~2000），用完再取。依赖DB但性能高。
2. Leaf-snowflake（雪花模式）：用ZooKeeper做workerId分配+时钟校验，解决雪花算法的时钟回拨问题。

面试亮点：提到美团在双11用Leaf-segment扛了每秒百万级ID生成。

Q6：Redis也能生成分布式ID，和雪花算法比怎样？

Redis方案：INCR order:id 原子自增。

对比：
- Redis更简单，一行命令就搞定
- 雪花算法不依赖外部服务，纯内存计算
- Redis方案需要持久化（RDB丢失会导致ID重复）
- 高并发下Redis是单点瓶颈（集群模式有切换延迟）

选择：QPS < 1万用Redis，QPS > 1万用雪花算法。

Q7：数据库号段模式（Segment）怎么避免号段丢失？

问题：取号段1001~2000→服务挂了→ID浪费。

解法：双Buffer优化。
Buffer1: 当前号段 [1001-2000]  用完后切Buffer2
Buffer2: 备选号段 [2001-3000]  异步加载下一个号段 [3001-4000]
号段快用完时（如用掉80%），异步线程提前加载下一个号段。切换时无缝衔接。

Q8：趋势递增和严格递增有什么区别？为什么雪花算法只是趋势递增？

严格递增：后一个ID一定比前一个大（数据库自增）。
趋势递增：整体递增，但相邻两个ID不一定谁大（雪花算法）。

为什么雪花算法只是趋势递增？
两个不同机器（workerId不同）在同一毫秒内生成ID——workerId小的机器生成的ID反而大。

这有问题吗？没有问题。MySQL InnoDB B+Tree索引写入不要求严格递增，趋势递增足够好，页分裂已经大幅减少。

Q9：雪花算法workerId怎么分配？

4种分配方式：
1. 配置文件写死：适合机器少且不变。简单但没法动态扩缩。
2. 数据库自增：启动时插一条记录拿ID。依赖DB但简单可靠。
3. ZooKeeper临时节点：启动时创建临时顺序节点拿编号，挂了自动释放。
4. Redis incr：跟DB类似，更快。

生产推荐ZooKeeper——美团Leaf-snowflake就是这个方案。

Q10：如果让你设计一个分布式ID系统，你会考虑哪些点？

面试终极追问——用这个框架答：
1. 全局唯一：雪花算法保证不重复。
2. 高性能：本地生成，不经网络，单机QPS百万+。
3. 高可用：不依赖外部服务（雪花模式），依赖服务要有降级方案（号段模式需DB主从）。
4. 趋势递增：方便数据库索引。
5. 可扩展：workerId能动态分配。
6. 防时钟回拨：启动时记录时间到磁盘，启动检查。

实习生说前4点就够了。后两点是加分。
