Q1：Cache-Aside、Read-Through、Write-Through、Write-Behind四种缓存策略分别是什么？

Cache-Aside（旁路缓存，最常用）：
• 读：先查缓存→命中返回→未命中查DB→写缓存→返回
• 写：更新DB→删除缓存（或更新缓存）
• Redis+MySQL经典组合就是Cache-Aside
• 优点：简单灵活，缓存故障不影响DB
• 缺点：代码入侵，需要手动管理缓存逻辑

Read-Through：缓存层自动读DB。应用只和缓存交互，缓存miss时自动load。
Write-Through：缓存层同时写缓存+DB，保证强一致。
Write-Behind（异步回写）：先写缓存，异步批量写DB。性能最高但数据可能丢失。

面试时一句话总结：Cache-Aside手动挡最灵活，Write-Through自动挡强一致，Write-Behind异步高性能。

Q2：缓存和数据库双写一致性怎么保证？先删缓存还是先写数据库？

核心方案：延迟双删
1. 先删除缓存
2. 更新数据库  
3. 延迟一段时间（如500ms）
4. 再次删除缓存
为什么不能先删缓存再更新DB？
A删缓存→B查缓存miss→B查DB（旧值）→B写缓存→A更新DB→缓存里是旧值，数据不一致。

为什么不能先更新DB再删缓存？
A查缓存miss→A查DB（旧值）→B更新DB→B删缓存→A写缓存（旧值）→数据不一致。但概率极低（A写缓存需要发生在B删缓存之后），所以实际大多用先更新DB再删缓存。

Canal+MQ方案（阿里常用）：监听MySQL binlog→发MQ→消费端删缓存。准实时，一致性强。

Q3：布隆过滤器原理？怎么解决缓存穿透？

布隆过滤器：一个bit数组+k个哈希函数。插入元素时把k个哈希位置都置1。查询时只要有一个位置是0就不存在，全是1则可能存在（有误判率）。

解决缓存穿透：把所有合法Key提前灌入布隆过滤器。查询时先过布隆→不存在直接返回空→存在则查缓存→缓存miss查DB。

Guava实现： 
BloomFilter filter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1000000,  // 预计元素量
    0.01      // 误判率
);
filter.put("userId:123");
if (!filter.mightContain(key)) return null;
Redis布隆过滤器：Redis 4.0+内置 bf.add / bf.exists 命令，比Guava更分布式友好。

Q4：缓存预热怎么做？秒杀场景下什么时候预热？

三种预热方式：
1. 启动预热（@PostConstruct/ApplicationReadyEvent）：项目启动时加载热点数据到Redis
2. 定时预热（@Scheduled）：每天凌晨重新加载最新数据
3. 触发预热（管理接口）：运营后台点"预热"按钮，手动触发

秒杀预热方案： 
@EventListener(ApplicationReadyEvent.class)
public void preheat() {
    List hotProducts = db.queryHotSeckill();
    for (Product p : hotProducts) {
        redis.set("seckill:stock:" + p.getId(), 
                   String.valueOf(p.getStock()));
    }
}
预热时机：秒杀开始前1小时执行预热，提前把库存和商品信息加载到Redis，避免秒杀瞬间缓存全部miss打崩DB。

Q5：多级缓存架构怎么设计？JVM本地缓存+Redis分布式缓存的协作？

典型三级架构：
Nginx缓存 → JVM本地缓存(Caffeine) → Redis → MySQL
查询流程：
1. 先查Caffeine本地缓存（最快，纳秒级）
2. miss→查Redis（次快，毫秒级）
3. miss→查MySQL（慢，秒级）→回写Redis→回写Caffeine

本地缓存选型：
• Caffeine：Spring Boot 3内置，W-TinyLFU算法，淘汰率高命中率
• Guava Cache：老牌，LRU淘汰

一致性方案：DB更新→删Redis→发MQ广播→各节点清本地缓存。或用短TTL（本地缓存30秒过期）+Redis主动失效兜底。

Q6：热点Key怎么探测和处理？

探测方式：
1. Redis --hotkeys：redis-cli --hotkeys，扫描所有Key统计访问频率
2. 客户端统计：在代码里对每个Key的访问计数，定时上报
3. 代理层：在Redis Proxy层统计，如Codis/Twemproxy

处理方案：
• 本地缓存兜底：把热点数据缓存到Caffeine，减少Redis压力
• Key拆分：将 product:123 拆成 product:123:0 到 product:123:9，分散到不同Redis节点
• 读写分离：热点Key走从节点读，分摊主节点压力
• 限流保护：对热点Key加令牌桶限流，保护下游

Q7：Redis内存满了怎么办？淘汰策略怎么选？

8种淘汰策略：
| 策略 | 行为 | |------|------| | noeviction | 不淘汰，报错 | | allkeys-lru | 所有Key中LRU淘汰 | | allkeys-lfu | 所有Key中LFU淘汰 | | allkeys-random | 随机淘汰 | | volatile-lru | 有TTL的Key中LRU淘汰 | | volatile-lfu | 有TTL的Key中LFU淘汰 | | volatile-random | 有TTL的Key中随机 | | volatile-ttl | 挑TTL最近的淘汰 | 选型建议：
• 缓存场景→allkeys-lru（淘汰最久没用的）
• 有热点数据→allkeys-lfu（保留高频访问的）
• 有持久化+过期→volatile-lru（只淘汰有过期时间的）

面试加分：提到LFU的logarithmic counter——不是精确计数，而是对数增长，节省内存。

Q8：Redis Cluster的16384个槽位怎么分配？为什么是16384？

槽位分配：CRC16(key) % 16384 决定Key落在哪个槽。
为什么是16384（16K）？
• CRC16结果范围是0~65535（64K）
• 如果用64K→心跳包太大（每个节点需告知所有槽位，bitmap就是8KB）
• 16K→心跳包2KB，控制在小包范围内
• 16K对集群规模（<1000节点）绰绰有余

迁移流程： 
CLUSTER SETSLOT 1234 MIGRATING nodeA
CLUSTER SETSLOT 1234 IMPORTING nodeB  
# 逐个迁移Key
MIGRATE host port key 0 5000
# 迁移完成
CLUSTER SETSLOT 1234 NODE nodeB

Q9：Redis Pipeline和事务(MULTI/EXEC)有什么区别？

Pipeline（管道）：
• 批量发送命令，减少RTT
• 不保证原子性——命令之间可能插入其他客户端的命令
• 适合批量读写（如一次性查100个Key）
• 性能提升：100个命令从100次RTT→1次RTT

事务（MULTI/EXEC）：
• 命令排队，EXEC一次性执行
• 保证原子性：事务内命令不会被其他客户端打断
• 不支持回滚：某条命令失败，其他命令继续执行
MULTI
SET user:1 "Alice"
INCR counter
EXEC
核心区别一句话：Pipeline优化网络，事务保证原子性。Lua脚本两者兼有（原子+批量）。

Q10：Redis大Key和热Key怎么排查和处理？

大Key发现：
• redis-cli --bigkeys：扫描所有Key，统计大小
• MEMORY USAGE keyname：查单个Key内存占用
• 业务侧：避免用Hash存储几十万字段、避免Set/Lsit无限增长

大Key处理：
• Hash→拆成多个小Hash（hash_tag路由到同一节点）
• List→拆成多个小List
• 删除用 UNLINK（异步删除）替代 DEL

热Key处理（见Q6）：
• 本地缓存兜底
• Key拆分到多节点
• 读写分离

面试金句："大Key用UNLINK异步删，热Key用本地缓存+Key拆分。Redis的单线程模型决定了热Key就是系统瓶颈，必须用'空间换时间'。"
