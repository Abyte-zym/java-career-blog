Q1：什么时候需要分库分表？判断标准是什么？

三个维度判断：
1. 数据量：单表超过500万-1000万行或超过2GB——查询开始变慢
2. 并发量：单库连接数超过1000——数据库扛不住
3. 磁盘：单机磁盘快满了

口诀："数据量大了分表，并发高了分库，磁盘满了都分。"
面试加分：提到先做缓存→索引优化→读写分离→最后才分库分表。分库分表是最后手段。

Q2：水平分库和垂直分库有什么区别？

垂直分库（按业务拆）：用户库、订单库、商品库各拆开。每个库的表结构不同。
水平分库（按数据行拆）：订单表太大→拆成order_0、order_1……每个表结构一样。

一句话：垂直拆业务（不同表放不同库），水平拆数据（同一表拆多份）。
实践：先垂直再水平——电商先拆用户/订单/商品三库，订单量大了再对订单表水平拆分。

Q3：分片键（Sharding Key）怎么选？

选择原则：
1. 高频查询条件：选最常出现在WHERE里的字段
2. 数据均匀分布：避免热点（如按userId比按sex均匀）
3. 唯一性：最好能唯一标识一行

反面案例：订单表按status分片→80%查询都是"待支付"，全打到同一片=无效分片。
正面案例：按user_id % 16哈希分片，数据均匀，查询单用户订单只需路由到一片。

Q4：ShardingSphere怎么配置分库分表？

Spring Boot配置示例： 
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1
      ds0: ...
      ds1: ...
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds$->{0..1}.t_order_$->{0..1}
            database-strategy:
              standard:
                sharding-column: user_id
                algorithm: inline
                algorithm-expression: ds$->{user_id % 2}
            table-strategy:
              standard:
                sharding-column: order_id
                algorithm: inline
                algorithm-expression: t_order_$->{order_id % 2}
解读：2库×2表=4个分片。user_id%2决定进哪个库，order_id%2决定进哪个表。

Q5：分库分表后跨库JOIN怎么处理？

6种解决方案（按推荐度排序）：
1. 字段冗余：把常用关联字段存多份——如订单表存user_name，不用JOIN用户表
2. 应用层组装：查两次——先查订单，拿到user_id列表，再批量查用户表，代码里拼
3. 全局表：小表（如配置表）每个库都存一份
4. 异构索引：ES同步宽表数据，用ES做关联查询
5. ShardingSphere绑定表：order和order_item用同一个分片键，保证在同一分片
6. 禁止跨库JOIN：这是最诚实的答案——分库分表后别想JOIN，从设计上规避

Q6：分布式全局ID怎么生成？雪花算法详解

雪花算法（Snowflake）结构：
1bit(不用) | 41bit时间戳 | 10bit机器ID | 12bit序列号

优势：趋势递增（索引友好）、全局唯一、不依赖数据库、每秒可生成409.6万个
时钟回拨问题：机器时间倒退→可能生成重复ID。解决方案：
- 记录上一次生成的时间戳，发现回拨抛异常或等待
- 美团Leaf用号段模式：每次从数据库取一段ID，用完再取，不依赖时钟
- 面试时说："我们项目用ShardingSphere内置的雪花算法，时钟回拨问题它已处理"

Q7：分库分表后怎么保证事务一致性？

三个层级：
1. 同一分片内：MySQL原生事务直接保证（数据在同一库同一表）
2. 跨分片：用Seata AT模式（自动回滚），或RocketMQ事务消息（最终一致）
3. 跨服务：用TCC（Try-Confirm-Cancel）或SAGA

面试这样说："分库分表后我们尽量避免跨分片事务。必须跨的用Seata AT模式——它对业务代码侵入小，性能损失可接受。"

Q8：读写分离怎么实现？主从延迟怎么处理？

实现方式： 
ShardingSphere 配置：
readwrite-splitting:
  data-sources:
    ds_0:
      write-data-source-name: ds0_master
      read-data-source-names: ds0_slave0, ds0_slave1
主从延迟处理：
1. 写后立即读：强制走主库（HintManager设置）
2. 对延迟敏感的业务：全走主库
3. 对延迟不敏感：走从库（如历史订单查询）
核心原则：能接受秒级延迟的走从库，不能接受的走主库。

Q9：数据迁移怎么搞？不停机迁移方案？

停机迁移（小厂）：
凌晨2点停服→导出SQL→分片工具预分片→导入新库→验证→上线
不停机迁移（大厂）：
1. 双写阶段：新老库同时写
2. 历史数据迁移：后台任务慢慢搬
3. 灰度切读：先切1%读流量到新库→监控→逐步放量
4. 下线老库：确认新库稳定后停写老库
工具：ShardingSphere-Scaling、Canal+自研脚本

Q10：分库分表后怎么分页查询？

问题：数据分散在4个库16个表里，LIMIT 100,10怎么办？

方案一（ShardingSphere已实现）：每个分片取LIMIT 0,110→合并排序→内存取第100-110条
⚠️ 越往后翻性能越差（每片都要取前面所有数据）

方案二（游标分页）：前端传lastId和pageSize，WHERE id > lastId LIMIT 10
✅ 性能稳定，但只能"下一页"，不能跳页

方案三（ES异构）：分页需求复杂→数据同步ES→ES做分页查询
面试技巧："分库分表后不做深度分页。产品设计上引导用户用搜索而非翻页。"
